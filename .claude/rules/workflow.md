# Workflow & Definition of Done

## Definition of Done (every feature, no exceptions)

1. Build succeeds (all targets)
2. Unit tests + UI tests written and passing
3. **Code review checklist**: no force unwraps, no prohibited patterns (Combine, SceneKit, AnyView, `@unchecked Sendable`), no parameterless `.animation()`, no `.accessibilityLabel()` on buttons (breaks XCUITest matching — use `.accessibilityHint()`), disabled buttons have `accessibilityHint` explaining WHY disabled
4. Update CLAUDE.md with any new patterns, gotchas, or anti-patterns discovered during implementation
5. Only THEN mark the task as complete

## R-CODIFY-EVERYTHING — anything that needs codification MUST be codified before close-out (2026-07-19)

**If a finding, lesson, gotcha, decision, convention, or failure-mode surfaced this session is DURABLE + REUSABLE (it will matter to a future session), it MUST be codified into the appropriate CANONICAL surface before the session closes — never left only in conversation, a task list, a PR body, or even a session-handoff doc.** A lesson that lives only in ephemeral context decays: the next session doesn't read this session's transcript or handoff, it reads the rules / CLAUDE.md / ADRs / templates. Codified per founder-direct 2026-07-19 (*"anything that needs codification needs to be codified"*). This generalizes DoD item 4 (CLAUDE.md) + Cross-Repo Audit Methodology Rule 5 (audit-to-canonical-propagation) into a universal close-out discipline.

**The test — "does this need codification?"** Yes if it is: a non-obvious gotcha that cost real time (a deploy-breaker, a merge trap, a scan-correctness bug); a reusable convention/pattern/workflow; a decision with alternatives worth not re-litigating (→ ADR); a failure-mode + its fix; a rule clarification or a newly-load-bearing invariant. No if it is purely tactical/one-off (a typo fix, a single commit's detail) or already fully covered by an existing rule (then at most add a one-line reference, don't duplicate).

**Where it goes (pick the canonical surface — a handoff doc is NOT one of them):**
- A **standing rule / invariant / gotcha** → the relevant `.claude/rules/*.md` (the surface loaded into EVERY session). Sync to app repos via `copy_rules_to_repos.sh` under the single-flight lock when portfolio-wide.
- A **decision with alternatives** → an ADR (`Docs/ADR-NNN_*.md`) + the ADR index.
- An **implementation convention the consuming session needs** → the relevant `Docs/TEMPLATE_*.md` / playbook / runbook.
- A **project-specific non-obvious fact** → `CLAUDE.md` (repo) or the file-based memory (per the memory guidance).
- The **research/plan/audit artifact itself** → `Docs/<KIND>_*.md` (per § "Save Research, Plans, and Audits").

**The discipline:** at close-out, ask *"what did I learn this session that a future session would need — and is it in a canonical surface yet?"* If not, codify it (a small dedicated PR is fine) BEFORE filing the handoff — the handoff then POINTS at the codified rule, it does not SUBSTITUTE for it. A session that discovers a durable lesson and closes without codifying it has an incomplete close-out. (Reference: the 2026-07-19 multi-band session codified 3 findings — node_modules-symlink deploy-breaker, css-parse-is-a-build-gate, spawned-app-DN-S-cast-pre-reserved — as rule edits, not just handoff prose.)

**Cross-refs:** DoD item 4 (CLAUDE.md) · § Cross-Repo Audit Methodology Rule 5 (audit-to-canonical-propagation) · § "Save Research, Plans, and Audits to Docs/" · § R-CANONICAL-DOC-GROUND-TRUTH (keep the codified canon true to on-disk reality) · the ADR convention (`Docs/ADR_INDEX.md`).

## 🔴 CRITICAL: Session Definition of Done — close-out checklist (every substantive session) — R-SESSION-CLOSE-OUT (2026-07-19; made CRITICAL 2026-07-20)

**A substantive session is NOT done when the last feature merges — it is done only after the close-out sequence below. Run these steps, in order, before declaring the session closed.** Codified per founder-direct 2026-07-19 (*"add workflow step to session handoff with remaining work items and priorities and with clean up and close out steps too"*). This is the SESSION-level DoD (the top-of-file DoD is per-FEATURE); a session that ships work but skips the handoff / clean-up / close-out verification has an incomplete close-out.

**🔴 CRITICAL (founder-direct 2026-07-20 — *"codify and make it critical,"* after the founder had to ask *"where is session handoff"* at the end of a big spawn session): filing the session handoff (step 2) is a HARD, NON-WAIVABLE close-out gate — on the same footing as "verify a PR merged before claiming SHIPPED." A substantive session that ends without a `Docs/CONTEXT_HANDOFF_<date>_<topic>.md` doc is INCOMPLETE, full stop. Never write the final user summary (step 5), and never declare the session closed, without having filed the handoff FIRST. The chat summary is not a substitute (it scrolls away; the handoff is what the next session reads). If you catch yourself about to summarize without a handoff, STOP and file it.**

Do this FIRST so the handoff can point at the codified rule.

A substantive session that ends with only a chat summary and no `CONTEXT_HANDOFF` doc has an incomplete close-out. The doc contains, at minimum:

   - **What turned out already-covered** — verified; duplicates discarded per § R-PARALLEL-HUB-AGENTS §5b — so the next session doesn't re-chase it.

   - **⚠ REMAINING/FOLLOW-ON work gets its OWN dedicated OPEN `## V<N>` entry — NEVER buried as inline sub-notes inside a SHIPPED entry or only in the handoff doc (R-WORK-QUEUE-OPEN-PICKUP; 2026-07-20).** The queue's PICKUP UNIT is the `## V<N>` header — a resuming session scans those headers for the next open item; it does NOT read the sub-bullets of a `V<N> — … SHIPPED` entry, nor the handoff, looking for work. So a follow-on recorded only as a "Follow-on (queued): …" bullet inside the shipped entry (or only in the handoff/audit doc) is effectively invisible as a pickup — it looks done. The SHIPPED record (what landed) and the OPEN follow-on pickup (what's left) are DISTINCT entries: write the shipped `## V<N>` for what landed, AND a separate **`## V<N+1> — OPEN`** for the remaining work (prioritized, pointing at the durable artifacts — audits, committed self-updating registries — that hold the detail). A self-updating registry / audit doc IS a fine home for the *detail*, but it still needs an OPEN `## V<N>` entry indexing it, or the next session won't find it. 3. **Clean up** (every repo the session touched):

   - `git checkout main && git pull --ff-only && git branch -D <merged-feature-branch>` per repo (§ Auto-Cycle return-to-main); confirm `git branch --show-current` = `main`.
   - Remove throwaway `git worktree`s (`git worktree remove … --force` your OWN named paths only, then `git worktree prune`; never a broad `wt-*` glob — § R-PARALLEL-HUB-AGENTS §9).
   - No stray uncommitted/untracked files left in a shared clone (`git status --porcelain` — pathspec-discard build artifacts, never `git add -A`).
4. **Close-out verification** (prove it, don't assume):
   - `gh pr list --state open --author "@me"` per repo = none (or intentionally-IN-FLIGHT, noted in the handoff) — a PR you think merged but shows OPEN is the orphan-PR trap (§ "Verify PR Merged").
   - Live-verify anything claimed shipped (browser-UA `curl -sL` = 200 for a route; registry row flipped only AFTER live-verify per R-WEB-CLONE-NO-DARK-SURFACE).

5. **Only THEN** state the session closed — and **ALWAYS end with a SESSION SUMMARY to the user** (a mandatory step, never skipped, founder-direct 2026-07-20): what shipped (verified), what's still open + its priority, and any account-managed/founder follow-ups. Report outcomes faithfully — if a step was skipped, a gate failed, or a lane yielded, say so plainly. The summary is the user-facing counterpart to the handoff doc (step 2): the handoff is the durable artifact for the next session, the summary is the turn's closing report to the founder. Even a non-substantive turn that doesn't need the full steps 1–4 still ends with a brief summary of what was done.

**When it applies:** any session that shipped code/content/docs or ran a multi-step task. A trivial one-turn Q&A or a single tactical fix doesn't need the full sequence (no remaining-work handoff for a closed one-liner) — but ANY session that leaves work unfinished, discovered a durable lesson, or ran in the parallel-fleet MUST do steps 1–4. **Cross-refs:** § R-CODIFY-EVERYTHING (step 1) · § "Save Research, Plans, and Audits to Docs/" · § Auto-Cycle Default (return-to-main) · § "Verify PR Merged Before Claiming SHIPPED" · § R-PARALLEL-HUB-AGENTS §5g (claim release) + §9 (worktree cleanup) + §1 (work-queue numbering) · `.claude/rules/spark-anvil-website.md` § R-WEB-CLONE-NO-DARK-SURFACE (live-verify before registry flip). ## Branching

- Feature branches off `main`, named `feature/<kebab-case-description>` (e.g., `feature/bar-model-canvas-accessibility`)
- One PR per feature branch, merged to `main` via merge commit (preserves branch history)
- Delete feature branches after merge — don't accumulate stale branches
- Keep feature branches current: merge `main` into the feature branch before PR to resolve conflicts
- Checkpoint commits on feature branches are encouraged — fine-grained history is squashed or preserved at PR merge time
- Push feature branches to `origin` for PR creation: `git push -u origin feature/<name>`
- When hub distributes shared rule updates while a feature branch is active, merge `main` into the feature branch to pick them up

## Auto-Cycle Default (Multi-Commit Work)

**For multi-commit / multi-round work in hub (and any session-recurrent autocycle pattern the user has established), the DEFAULT loop is `branch → commit → push → gh pr create → gh pr merge --merge --delete-branch → verify → return-to-main` — no per-step confirmation prompts.** This applies when:

- The user has previously approved an autocycle pattern in the session (e.g., "go with your recs", "pre-approved", "start next round")
- The change is research / documentation / queue update / planning (low-risk; reversible via git)
- The PR is in a hub-owned repo (hub / spark-anvil-site per Round 76 scope-rule change)

**Do NOT auto-cycle for**:
- Cross-repo PRs touching app-repo Swift source code (still scope-rule-bound; app sessions own implementation)
- Force-pushes, --no-verify, rebase --interactive, or any destructive op
- Production deployment / DNS / hosting changes
- First-time scope-rule changes (those need explicit user "go")

Codified in memory `feedback_branch_workflow.md`. The verify step (`gh pr view <n> --json state,mergedAt`) is REQUIRED per the rule below; auto-cycle does not skip verification.

**`--delete-branch` deletes only the REMOTE branch — the local clone STAYS on the merged feature branch. `return-to-main` is a required round-close step, per repo touched.** `gh pr merge --delete-branch` removes `origin/<branch>` but does NOT move your local `HEAD` or delete the local branch — so after a merge the working clone is still sitting on a now-merged (and remotely-deleted) feature branch. Leaving it there is a stale-state trap: the next round branches off the wrong base, a `git pull` looks like a no-op against the deleted upstream, and freshness/pre-work checks read the wrong tree. So the loop's final step, **for every repo a round touched**, is:

```bash
git checkout main && git pull --ff-only && git branch -D <merged-feature-branch>
```

Verify with `git branch --show-current` → `main` in each repo before declaring the round done. (Reference: 2026-07-13 — a completed scienceforge clone left all-merged but the hub clone still on its feature branch until asked "why are we not back on main yet.") This composes with § "Per-repo pull-then-audit BEFORE work" (that's the inbound side; this is the outbound round-close) and applies equally to worktree builds — remove the throwaway worktree (`git worktree remove`) as its analog of return-to-main.

**GATE the local `git branch -D` on a VERIFIED-merged state — never chain it unconditionally after the merge in the same block.** Newline-separated commands in one shell block run regardless of the previous command's exit code, so a `gh pr merge` that FAILS (a parallel-fleet merge-race conflict → `not mergeable`, the routine `play.css`-union case) followed by an unconditional `git checkout main && git branch -D <feature>` will **delete your only local copy of the branch while the PR is still OPEN** — turning a 10-second re-merge into a `gh pr checkout` / remote-branch-recovery detour. So the round-close order is strict: `gh pr merge` → **`gh pr view --json state` must read `MERGED`** (the § "Verify PR Merged" gate) → **only then** `git checkout main && git pull --ff-only && git branch -D <feature>`. If the merge didn't land (CONFLICTING/BLOCKED/UNSTABLE), STOP at the verify step, rebase-and-retry, and do NOT run the branch-delete. The remote PR branch (and thus your work) survives a botched local cleanup, but recovering it costs a detour you avoid by gating. (Reference: 2026-07-16 — a tinyletters hub-PR merge hit a work-queue-`V<N>` conflict; the same block's chained `git branch -D` deleted the local branch mid-race, needing a remote-branch recovery before the rebase-resolve could proceed.)

**The SAME unconditional-chaining trap bites the FRONT of the chain too — after a history-rewriting `git rebase`, a plain `git push` FAILS non-fast-forward, and a chained `gh pr create` then silently builds the PR from the STALE origin branch (pre-rebase → missing commits → CONFLICTING + content-incomplete).** A rebase rewrites your local history so the remote branch is no longer an ancestor; `git push` (no flag) is correctly rejected ("use 'git pull' before pushing again"), but if you chained `push → gh pr create → gh pr merge → git branch -D` in one block, `gh pr create` builds from whatever is CURRENTLY on origin (the old pre-rebase tip, or nothing new), the merge fails/conflicts, and the branch-delete then discards your rebased local commits to a dangling state. So: (a) **after a rebase, push with `git push --force-with-lease`** (never a bare `git push`, never chain it); (b) **verify the push landed AND the PR head == your local HEAD** (`git rev-parse HEAD` == the PR's head sha, or just eyeball `gh pr view --json commits`) BEFORE `gh pr merge`; (c) then the line-45 verify-MERGED-before-delete gate. Do NOT run `push`/`gh pr create`/`gh pr merge`/`git branch -D` as one unconditional block across a rebase — check each step's result. (Reference: 2026-07-16 — a dancequest hub finish-PR: rebased locally, bare `git push` was rejected non-ff, the chained `gh pr create` built PR #1782 from the stale origin branch [CONFLICTING, missing the Track-C doc commit], and the chained `git branch -D` dropped the rebased commits; recovered cleanly via a fresh branch off current `origin/main` + re-applied edits, zero shipped-work loss, but a full detour the gate avoids.)

**`--force-with-lease` rejected with `stale info` on a WORKTREE branch that DIVERGED from the remote PR tip (a prior/parallel session pushed a newer commit onto the PR branch after your worktree was created) — do NOT blind-`--force`; compare against the FETCHED tip, verify your rebased HEAD is a proper up-to-date SUPERSET, then lease against the fetched sha (2026-07-21, agoraquest #1158).** A resumed/handed-off worktree's local branch can be STALE vs what's actually on the remote PR branch (the earlier session pushed a "6/6" commit your worktree never had), so a rebase-then-`--force-with-lease` fails `stale info` (the lease's cached remote ref predates the newer commit). The DANGER is a reflexive plain `--force` — it could clobber the remote's real PR content. The safe resolution: (a) `git fetch origin <branch>` — **under a narrow fetch refspec (a purpose-built worktree tracks only `origin/main`) the branch ref lands in `FETCH_HEAD`, not `origin/<branch>`**, so compare `git rev-parse HEAD` vs `git rev-parse FETCH_HEAD`; (b) `git merge-base --is-ancestor FETCH_HEAD HEAD` + `git diff --stat HEAD FETCH_HEAD` — CONFIRM your rebased HEAD is a proper up-to-date SUPERSET: the diff must show only your target files (byte-identical to the remote's on your clone's target) + files that are ALREADY-MERGED-on-main (which your rebase-onto-current-main legitimately has and the stale tip lacks), with ZERO unexpected reverts of the PR's own content; (c) only then `git push --force-with-lease=<branch>:<the FETCH_HEAD sha> origin HEAD:<branch>` (the explicit-sha lease is current because you just fetched it). Never plain `--force` blindly; never the default (bare) `--force-with-lease` here (its cached lease is the stale thing). Reference: the agoraquest worktree branch lacked the remote's `ae986e49` "6/6" commit; the `HEAD..FETCH_HEAD` diff proved the agoraquest files byte-identical (the only FETCH_HEAD-unique content was stale/pre-merge state my rebase-onto-current-main superseded), so the explicit-sha lease force was safe.

## CRITICAL: Verify PR Merged Before Claiming SHIPPED

**Never mark a PR "SHIPPED" until you have confirmed the merge.** This rule was codified after 3 orphan-PR incidents in hub (Round 70 #377 LiveKit DECISION on PR #208; Round 73 PR title-mismatch on PR #220; Round 76 #392 beta-testing surface on PR #86) where queue docs falsely claimed "SHIPPED" while the PR sat OPEN on its feature branch.

**Required verification after every PR creation**:

```bash
gh pr merge <number> --merge --delete-branch
# Then verify:
gh pr view <number> --json state,mergedAt
# State must be MERGED. mergedAt must be non-null.
```

If `gh pr merge` returns "Pull Request is not mergeable" (status `UNSTABLE`):
- Check what's blocking: `gh pr view <number> --json mergeStateStatus,statusCheckRollup`
- If it's a non-blocking CI check (Cloudflare Workers build IN_PROGRESS, optional lints), use `--admin` to merge
- If it's a real failure or merge conflict, fix before merging — do NOT close the round with the PR still OPEN

**A PR that shows "no checks reported" is almost always CONFLICTING, not "CI queued/broken" (2026-07-15).** GitHub does **not** run `pull_request` workflows on a PR whose merge ref can't be created, so when `origin/main` advances under you (routine with parallel sessions — § R-PARALLEL-HUB-AGENTS) and the branch conflicts, the PR sits at `mergeable=CONFLICTING` / `mergeStateStatus=DIRTY` and **zero checks ever appear**. Do NOT wait on CI or push empty "trigger" commits — **`gh pr view <n> --json mergeable,mergeStateStatus` FIRST**, then `git rebase origin/main`, re-run the repo's parse/build gates (a union-merged file — e.g. `play.css` — can auto-merge to *invalid* content with NO conflict marker, so the gate is what catches it), and force-push; CI triggers once the PR is mergeable again.

**⚠ BUT "no checks reported" is NOT always CONFLICTING — disambiguate by `mergeable`/`mergeStateStatus` FIRST (2026-07-21).** The rule above is the CONFLICTING case (`mergeable=CONFLICTING`/`DIRTY`). The OTHER "no checks reported" case is `mergeable=MERGEABLE`/`mergeStateStatus=CLEAN` with zero checks — which means the workflow simply **did not attach** to this PR (not a conflict): e.g. a `spark-anvil-site` PR touching ONLY a **backend/non-play path** the `play-tests` scope skips (`workers/**`, `src/lib/<non-play>/**` — NOT matched by the FULL-sweep `SHARED_RE` nor a clone route), AND `main` has **no required status checks** (verify: `gh api repos/<o>/<r>/branches/main/protection/required_status_checks` → "not accessible"/empty). In that case `CLEAN` is authoritative — GitHub says nothing is blocking — so **merging is safe** (don't over-wait for a run that will never attach); just satisfy the one relevant gate LOCALLY first (e.g. `npx vitest run <the-new-test>`). So the decision is: `mergeable=CONFLICTING` + no checks → rebase (above); `mergeable=MERGEABLE`+`CLEAN` + no checks + no required checks on main → workflow didn't attach → merge after a local gate. Reference: site #1169 (a `workers/efficacy/` + `src/lib/efficacy/` backend PR) sat MERGEABLE/CLEAN with no attached run; Vitest was verified locally (8/8) and it merged cleanly.

**Before closing any multi-PR round**:

```bash
# Audit any feature branches that haven't been merged
gh pr list --state open --author "@me"
```

If the queue claims SHIPPED but `gh pr list --state open` shows the branch, the round is NOT actually closed. Fix immediately by merging or marking IN-FLIGHT.

**Common failure mode**: bg agents create a feature branch + PR but the agent process ends before `gh pr merge` runs (or runs but check-blocks). The agent's report says "shipped" but the merge never happened. **The main session must verify** — agent reports are not authoritative for merge state.

**🔴 "auto-poll-merging" / "auto-merge in progress" / a launched background poll-and-merge loop is NOT a merge — a queue/handoff/CLAIMS line that records SHIPPED on that basis is FALSE-SHIPPED, and the PR sits OPEN + the route 404 indefinitely once that loop's session dies (R-AUTOMERGE-NOT-MERGED; 2026-07-21).** The § R-PARALLEL-HUB-AGENTS §4(e) escape sanctions launching a **background poll-and-merge loop** to land a PR once CI clears — but that loop lives INSIDE its session, so if the session is **killed/ends before `CLEAN`** (routine in the parallel fleet: rate-limit death, founder stop, harness end), the loop dies with it, the PR **never merges**, and any status the session already wrote ("SHIPPED — #NNNN auto-poll-merging", a flipped registry row, a work-queue `SHIPPED` header) is a **lie the next session inherits as truth**. This is the automation-shaped form of the orphan-PR trap: the record says done, origin says OPEN. **The discipline:** (1) **NEVER record SHIPPED — in the work queue, a handoff, `.claude/CLAIMS.md`, or a registry row — until `gh pr view <n> --json state` reads `MERGED` AND (for a user-facing route) the live URL returns 200 with real content.** "auto-merging"/"poll-loop-running"/"will merge when CI clears" is an IN-FLIGHT state → record it as **`#NNNN auto-merge PENDING`**, never SHIPPED. (2) A **resuming/close-out session MUST re-verify every "SHIPPED" claim from the prior session against origin** (`gh pr view` state + a live `curl`) — a background-merge claim is the highest-risk kind to trust. (3) When you inherit a false-SHIPPED PR, verify-locally-then-shepherd-to-merge (don't re-run the dead loop): confirm mergeable + gates green (a **cancelled** CI check is *not* a failure — re-run it or prove the equivalent locally, § R-PARALLEL-HUB-AGENTS §4(c)), merge, verify `MERGED`, live-verify, THEN flip the record. Reference incident: work-queue V483 recorded "TeachTales Phase B SHIPPED (#1172 auto-poll-merging)" while #1172 sat **OPEN + `/play/teach` 404 for ~1.5h** (the poll-loop died with its killed session; the a11y CI run was *cancelled* under pool saturation, not failed); a founder-authorized in-session verify (local Vitest+css-parse+build:play+dual-scheme screenshot-DoD green) shepherded it to a proven-green admin-merge (`458dc434`) + `.com`/`.org` 200 live-verify, then corrected the record.

## CRITICAL: Verify origin state before claiming portfolio-wide content coverage

**This rule extends "Verify PR Merged" to cover content distribution.** Wave runners that `cp` assets directly into a sibling repo's working tree (e.g., `t2_coverage_wave_runner.sh` → `spark-anvil-site/public/chapters/`) do NOT auto-commit. The operator MUST run a separate commit + push cycle, AND verify origin reflects the new state, BEFORE authoring any handoff doc that quotes coverage numbers.

**Codified after V10 (2026-06-23) — the 1497-file orphan-state incident**: V10's wave runner generated 252 Tier-2 chapters + distributed to the site working tree, but a mid-round P0 YAML drift incident pulled the operator's attention. The bulk content sat staged-but-unpushed while the V10 handoff doc claimed "252/252 Tier-2 audio drama m4a files at site, zero partials, 100% portfolio coverage" — true locally, false at origin (origin had 86). V11 Wave 1 had to ship a P0 sync push (spark-anvil-site PR #280) to close the gap before continuing.

### Required verification before authoring any handoff coverage claim

```bash
# 1. Fetch origin to ensure local view of origin is current
cd ../spark-anvil-site && git fetch origin main

# 2. Verify origin state matches local state for the path in question
git ls-tree -r origin/main public/chapters/ | grep -c '\-advanced_chapter\.m4a'  # origin count
git ls-tree -r HEAD public/chapters/ | grep -c '\-advanced_chapter\.m4a'         # local-tracked count

# 3. Check for uncommitted local state — MUST be empty before SHIPPED claim
git status --porcelain public/chapters/ | wc -l                                  # MUST = 0
```

If `git status --porcelain` shows ≥1 file in the distribution path, the round is NOT actually shipped. Either commit + push the pending state, OR change the handoff to mark IN-FLIGHT.

### Wave-runner self-report (enforcement at source)

All three wave runners (`t2_coverage_wave_runner.sh` / `path_b_wave_runner.sh` / `path_b_tier2_audio_wave_runner.sh`) end with a check that warns when the site working tree has uncommitted content in `public/chapters/` + `src/data/`. Codified V11 (2026-06-23). Sample output:

```
⚠  spark-anvil-site has 1497 uncommitted files in public/chapters/ + src/data/.
   Run cd ../spark-anvil-site && git status to inspect; commit + push before claiming coverage.
   Origin state ≠ local state; handoff docs MUST verify against origin.
```

Operators must heed the warning before round-close. The warning fires regardless of context-switch / interruption / etc. — it's the last line of defense against the V10 failure mode.

### When this rule applies

- Every wave runner that distributes assets to a sibling repo's working tree (current: 3 site wave runners; future: any new `gen_*_to_site.py` / `copy_*_to_repos.sh` script that produces site assets)
- Every handoff doc that quotes site coverage numbers (e.g., "X/Y chapters at site", "100% coverage")
- Every round-close that bundles cross-repo content distribution

## Stale-clone recovery — verify 3 safety conditions before `git reset --hard` (2026-07-02)

**When a clone shows a large "modified + untracked" working tree AND `git pull --ff-only` aborts, do NOT assume it's an orphan-state (unpushed local work) and do NOT blindly reset.** The far more common cause is a **stale local `main` pointer** — the branch is dozens/hundreds of commits behind origin with a superseded working tree, and the "untracked" files are already tracked upstream. Codified after the 2026-06-30 FractionForge session: the hub clone showed **822 "modified" + 467 untracked** files with `--ff-only` aborting; it turned out `main` was **563 commits behind** origin with a ~2026-06-11-era tree — zero real local work. This is DISTINCT from the V10 orphan-state rule above (that was genuinely *unpushed local content*; this is *stale pointer + superseded tree*).

**Before any `git reset --hard origin/main`, verify all 3 safety conditions — and STOP if any fails:**

```bash
git fetch origin main
# (a) No unpushed commits on HEAD (== 0)
git rev-list --count origin/main..HEAD          # MUST print 0
# (b) HEAD is a strict ancestor of origin/main (reset is a pure fast-forward, loses no commits)
git merge-base --is-ancestor HEAD origin/main && echo "ff-safe" || echo "DIVERGED — STOP"
# (c) Sampled "untracked" files already exist on origin (they're not new local work)
for f in <sample-untracked-path-1> <sample-untracked-path-2>; do
  git cat-file -e "origin/main:$f" 2>/dev/null && echo "$f: on origin (safe)" || echo "$f: NOT on origin — INVESTIGATE"
done
```

Only when (a) == 0 AND (b) is ff-safe AND (c) all sampled untracked files exist on origin → `git reset --hard origin/main` is safe (it discards a stale tree, not real work). If (a) > 0 or (b) reports DIVERGED or (c) finds a genuinely-new file, STOP and surface to the user — that IS unpushed work and needs the orphan-state handling, not a reset.

**Refinement 1 — a HISTORY REWRITE is the OTHER benign cause of (a) > 0 / DIVERGED; don't misread it as unpushed work (2026-07-17).** After a sibling session runs a history purge/rewrite (R-SITE-HISTORY-PURGE, or the hub's own ADR-047 purge), every recent commit is re-SHA'd, so `origin/main..HEAD` is LARGE by construction (observed: 1403) and `merge-base --is-ancestor` reports DIVERGED — even with ZERO genuine unpushed work. The "(a) > 0 → STOP, that IS unpushed work" gate above is too absolute for this case. **Distinguish a rewrite from real unpushed work by TWO signals:** (1) `git fetch origin main` prints **`(forced update)`** on the `main` ref (a plain advance never does); (2) your recent local commit *subjects* are present on origin under NEW SHAs — `git log -1 --format=%s HEAD`, then `git log origin/main --format='%h %s' | grep -F "<that subject>"` returns a hit with a different short-SHA. When BOTH hold and your last few commits are all published PR-merge/registry commits (no local authored-but-unpushed work), the divergence is a rewrite, not orphan-state → `git reset --hard origin/main` is safe. If a forced-update fetch does NOT reproduce your HEAD's subject on origin, that commit IS unpushed local work — STOP (orphan-state handling). Reference: 2026-07-17 hub clone reconciled after the ADR-047 purge rewrote SHAs twice (1403 "unpushed"/5220 behind/DIVERGED, but HEAD subject present on origin under a new SHA → clean reset). **⚠ The subject-grep FALSE-POSITIVES when a branch merged via a squash / PR-merge commit or was renumbered under the parallel fleet** — the individual commit's subject then does NOT appear verbatim on origin even though the work fully landed (2026-07-17: a stale `hub/web-clone-soundsphere` branch showed 2 "UNMERGED" subjects, but the clone had merged via `Merge pull request #1396` + been renumbered #38→#39, so `Docs/web/soundsphere/` + porter + registry row were all on origin). So when the subject-grep misses, do NOT conclude "unpushed" — confirm by checking the branch's **deliverable ARTIFACTS on origin** (`git ls-tree -r origin/main -- <the branch's output files>`, `git log origin/main --grep=<branch-name>` for its merge commit), i.e. R-PARALLEL-HUB-AGENTS §5a "grep origin for the deliverable FILE, not a status field." Only a subject-miss AND artifacts-absent is genuine unpushed work.

**Refinement 2 — `git reset --hard` does NOT reclaim local `.git` disk after a purge; re-clone or gc-prune to actually shrink it (2026-07-17).** Resetting to a purged/rewritten origin gives you the correct tip + a safe-to-push state, but the OLD pre-purge pack objects stay in your local store (pinned by the reflog + the just-fetched refs), so your local `.git` stays BLOATED even though origin is lean. Reference: post-reset the hub `.git` was still **7.2 GB** (`git count-objects -v` → `size-pack: 7350564`, 30 packs, 388 reflog entries) while origin was **119 MB**. Reset = *correct + pushable*, NOT *lean*. **⚠ `reflog expire --expire=now --all && gc --prune=now` ALONE under-reclaims — the dominant pin is usually STALE LOCAL BRANCHES (and stale remote-tracking refs) still pointing at PRE-purge history, which keeps the big purged blobs reachable so gc can't drop them (2026-07-17: gc alone only moved 7.2 GB → 6.8 GB; the store held 52,592 in-pack objects reachable from 4 abandoned `feature/*`/`hub/*` clone-doc branches left un-deleted at prior round-closes — each descended from pre-purge history carrying the 4.6 GB PDFBooks + 2.3 GB R2AudioBackup blobs).** So the reclaim order is: (1) `git for-each-ref refs/heads refs/remotes` + flag every ref NOT an ancestor of `origin/main`; (2) for each, verify its work is on origin (Refinement 1's **deliverable-artifacts** check — the subject-grep false-positives on merged-but-renumbered branches) and `git branch -D` / `git remote prune origin` the confirmed-safe ones; (3) THEN `git reflog expire --expire=now --all && git gc --prune=now`. **⚠ `git remote prune origin` will NOT drop a stale `origin/feature/*` tracking ref whose remote branch STILL EXISTS on the server — prune only removes refs whose upstream was *deleted* (2026-07-17).** An abandoned, already-merged remote feature branch that was never deleted at round-close (the exact lingering-branch hygiene gap) therefore survives prune AND keeps pinning pre-purge blobs locally, so gc can't reclaim. Fix: after confirming its work is on origin (deliverable-artifacts check), drop the **local tracking ref only** with `git branch -rd origin/feature/<name>` — this unpins the blobs, is fully reversible (a future full `git fetch origin` re-creates it), and does NOT touch the shared remote branch (deleting *that* is an outward-facing action on another lane's branch — flag it for the founder / owning session, don't do it as part of your local reclaim). Reference: 2026-07-17 the 16 GB→132 MB hub reclaim — `origin/feature/web-terrawatch-hubside` (terrawatch `shipped` + its V293 commit already on main) survived `remote prune`; `git branch -rd` on the local ref let gc finish the reclaim. After deleting the 4 stale branches this dropped the hub `.git` **6.8 GB → 132 MB** (≈ the 119 MB origin). Alternatively skip all of it with a fresh `git clone` (preferably `--filter=blob:none`, per `.claude/rules/spark-anvil-website.md` § R-SITE-BLOBLESS-CLONE), which has no stale refs by construction. A tiny-branch `git push` works fine from a bloated-but-correct store regardless (only reachable-not-on-remote objects are sent), so reclamation is disk-hygiene, not a push prerequisite — but a coordination message that says "reset --hard OR re-clone" understates that reset alone (even with gc) does NOT reclaim while stale pre-purge branches linger.

**Refinement 3 — heads + remotes are NOT the only gc roots; sweep EVERY ref namespace (esp. `refs/stash`) + `ORIG_HEAD`, and never launch a 2nd gc while one is running, or reclaim takes 3 passes instead of 1 (2026-07-17).** Refinement 2's reclaim order (flag non-ancestor `refs/heads`+`refs/remotes`, delete, gc) is necessary but INCOMPLETE — it misses three additional pre-purge-blob pins that each independently keep the big pack alive, so a gc after "delete stale branches" still under-reclaims:
  1. **`refs/stash`** — a leftover WIP stash is a real gc root whose parent chain reaches into pre-purge `main` → pins the entire old pack. **⚠ `git stash list` can print EMPTY while `refs/stash` still exists** (expiring reflogs — Refinement 2's `reflog expire --all` — drops the stash's *reflog* but NOT the `refs/stash` ref itself), so a "no stashes" reading is a false all-clear. Check the ref directly: `git for-each-ref refs/stash`. Before dropping, verify its content is on origin (`git stash show --stat <sha>` → for each file `git cat-file -e origin/main:<f>` + `git diff origin/main <sha> -- <files>` empty-or-superseded), then `git update-ref -d refs/stash`. Reference: an ancient (2026-04-15) 3-doc BridgeForge WIP stash — all 3 docs present + superseded on origin — pinned the whole 8.9 GB pack.
  2. **`ORIG_HEAD`** — `git reset --hard` WRITES `ORIG_HEAD` = the pre-reset (old pre-purge) tip, another gc root pinning old history. Clear it: `rm -f .git/ORIG_HEAD` (or `git update-ref -d ORIG_HEAD`).
  3. **Stale `refs/remotes/origin/*`** — `git remote prune origin` only drops refs that were DELETED on the remote **AND are within the clone's fetch-refspec scope**: under a NARROW refspec (`git config --get-all remote.origin.fetch` shows only `+refs/heads/main:refs/remotes/origin/main`, common on a purpose-built hub/site clone) both `remote prune` and `git fetch --prune` skip `origin/feature/*` entirely, so those refs survive EVEN when the branch was already deleted server-side (2026-07-17: 3 such refs, confirmed gone via `git ls-remote --heads origin`, still pinned the whole 6.97 GiB pack until `update-ref -d`'d — so post-reset gc stalled at 7.0 GB, then dropped to 116 MB once they were dropped). A branch still present on GitHub keeps its LOCAL remote-tracking ref at its PRE-purge SHA (you fetched only `origin/main`, so the rest never updated) — either way, don't rely on prune. Those are gc roots too. They are pure cache (a future `git fetch` repopulates them at correct post-purge SHAs), so delete all non-main: `git for-each-ref --format='%(refname)' refs/remotes/origin | grep -vE '/(main|HEAD)$' | while read r; do git update-ref -d "$r"; done`. Reference: 39 stale `origin/feature/*` refs survived `remote prune` and blocked the reclaim.

  **The one-pass reclaim checklist (do ALL before the single gc):** `git for-each-ref` (ALL namespaces — heads, remotes, tags, stash, notes) → for each non-`main` ref verify its work is on origin (Refinement 1 deliverable-artifacts check) → delete the confirmed-safe (`git branch -D` / `git update-ref -d` / `git worktree prune` for prunable worktrees) → `rm -f .git/ORIG_HEAD` → `git reflog expire --expire=now --all` → confirm the only refs left are `main` + `origin/main` (+`origin/HEAD`) → **then** one `git gc --prune=now`. **⚠ Never fire a 2nd `git gc` while one is still running** — it bails `fatal: gc is already running on machine … pid N`, AND a gc that STARTED before you dropped a root enumerated its reachable set at launch, so it repacks the old blobs INTO a fresh pack and won't drop them (that gc's output can briefly BALLOON `.git` — a mid-repack 9.1 GB→15 GB was observed — as the new temp pack coexists with the old). Confirm no gc is running (`pgrep -fl "git gc|git-pack-objects|git-repack"`) and all roots are gone, then run exactly one gc. Doing the full sweep up front took the hub `.git` **9.1 GB → 132 MB in the final pass** after two earlier wasted gcs (one predating the stash drop, one bailing on the pid lock). **⚠ Tail-cleanup: an interrupted / pid-locked gc leaves orphaned `.git/objects/pack/tmp_pack_*` files that persist as `size-garbage` and PHANTOM-re-bloat `.git` even though the real store is lean — a plain `git gc` does NOT remove recent temp-packs.** After the reclaim (and after any `git pull` that surfaces them), verify with `git count-objects -vH` (compare `size-pack` vs `size-garbage`): a lean `size-pack` (~120 MiB here) with a large `size-garbage` (2.3 GiB of two `tmp_pack_*`, dropping `.git` back to 132 MB once removed) means the store is fine and the bloat is pure garbage — `git fsck` names them "garbage found"; with no gc running, `rm -f .git/objects/pack/tmp_pack_*` and re-`fsck`. Don't mistake `size-garbage` for un-reclaimed history.

**Refinement 4 — on a reclaimed clone, a BARE `git pull` / `git fetch` RE-DOWNLOADS the pre-purge blobs and RE-BLOATS `.git`; use TARGETED `… origin main` for freshness until the stale origin branches are deleted server-side (2026-07-17).** The purge rewrote ONLY `main` — the old already-merged feature branches that were never deleted at round-close **still exist on origin at their PRE-purge SHAs**. So the default fetch refspec (`+refs/heads/*:refs/remotes/origin/*`, which a no-arg `git pull` / `git fetch` / `git fetch --all` uses) re-fetches *all* of them, re-downloading the exact purged pack you just reclaimed and re-creating the stale `origin/feature/*` gc-roots. This **sharpens Refinement 3.3's "a future `git fetch` repopulates them at correct *post-purge* SHAs"** — that holds ONLY for a REWRITTEN ref; an unrewritten still-on-origin branch repopulates at its PRE-purge SHA and re-bloats. Reference: 2026-07-17, seconds after the 9.2 GB → 132 MB reclaim, a habitual no-arg `git pull --ff-only` started an `index-pack` of **41,608 objects** and took `.git` **132 MB → 2.6 GB** before it was killed; the same freshness via `git pull --ff-only origin main` (targeted) fast-forwarded with a tiny pack and left `.git` lean. **Discipline on a reclaimed clone: fetch freshness with `git fetch origin main` / `git pull --ff-only origin main` (targeted) — NEVER a no-arg `git pull` / `git fetch` / `git fetch --all`** — until the lingering origin branches are actually deleted server-side (an outward-facing action on other lanes' branches → founder-authorized, not part of your local reclaim). **The DURABLE fix (not just the local workaround) is to SWEEP those stale origin branches — a purge's required completion step, each verified DELETE-SAFE by the post-rewrite subject-presence + deliverable-artifact method, per `portfolio.md` § R-GIT-BLOAT-HYGIENE invariant 7.** Until they're gone, EVERY fresh clone re-downloads the pre-purge pack, not just yours (2026-07-17: a *fresh* hub clone was 8.9 GB solely because 38 already-merged branches were left on origin post-ADR-047 — sweeping them under a founder GO took fresh-clone size to ~115 MB). A blobless re-clone (§ R-SITE-BLOBLESS-CLONE) sidesteps the LOCAL symptom by construction but does NOT fix origin for the next cloner. If a bare fetch already re-bloated you, its download is unreferenced under a targeted refspec → re-run the Refinement 3 one-pass reclaim (drop any re-created `origin/*` refs, `git gc --prune=now`, sweep `tmp_pack_*`). **Two complements to the targeted-fetch discipline:** (1) **decisive diagnostic to confirm the pin is non-`main` refs before acting** — `git rev-list --disk-usage --objects origin/main` vs `git rev-list --disk-usage --objects --all`; a huge gap (observed **114 MB vs 9151 MB**) proves non-`main` refs hold the bloat (and that dropping them + gc will reclaim). (2) **a stronger durable fix than remembering to type `origin main` on every fetch — convert the clone to blobless IN PLACE** (per `.claude/rules/spark-anvil-website.md` § R-SITE-BLOBLESS-CLONE): `git config remote.origin.promisor true && git config remote.origin.partialclonefilter blob:none` — then even a bare `git fetch`/`pull` pulls commit+tree objects only (blobs stay lazy on origin) and CAN'T re-bloat the history; caveat: deep-history ops (`git log -p` far back, checking out an old commit) then lazily re-fetch blobs (needs network) — fine for main-based PR work. Reference: the 2026-07-17 reclaim set this config, and the next `git pull` (integrating a sibling push) stayed lean instead of re-inflating.

## CRITICAL: Stagger Background Agents — Never 4+ in Parallel

**Spawn background agents one at a time (sequentially), not 4+ simultaneously.** Codified after a Round 90 (2026-05-27) incident where 4 parallel `run_in_background: true` agents all hit the Anthropic API rate-limit cap within ~12 minutes wall-clock with zero/minimal work completed across the wave. Round 91 (same day) re-ran the same 4 work items staggered (1 at a time) and shipped all 4 successfully — total ~552K agent tokens, no rate-limit caps.

### Empirical pattern

| Strategy | Outcome | Tokens spent |
|---|---|---|
| **4 bg agents parallel** (Round 90) | ALL 4 capped within 12 min; #466 hygiene got furthest at 2,999 tokens / 59 tool calls — others 400-1,000 tokens each | ~5K total before cap |
| **1 bg agent at a time** (Round 91) | 3 of 3 completed cleanly: #468 P0 fix (180K tokens / 48 tools / 54 min) + #469 Wave D (272K / 74 / 34 min) + #470 Gemma research (75K / 41 / 14 min) | ~552K total, no caps |

### Recommended workflow

For multi-bg-agent rounds:

1. **Spawn agent #1** in background via `Agent({run_in_background: true})`
2. **Do main-session foreground work in parallel** — independent docs, queue updates, drift fixes, planning. Main session and bg agent run on separate token budgets
3. **Wait for the `<task-notification>` completion message** for agent #1 (do NOT poll; the system notifies)
4. **Spawn agent #2** once #1 completes
5. **Repeat** for #3, #4, etc.

### Why parallel saturates

Background agents share the same Anthropic API limit pool as the main session. Each agent's first turn does discovery (reads, greps, doc surveys) consuming 5-20K tokens. Four parallel discoveries = 20-80K tokens within minutes, saturating the limit before any productive work lands. Sequential spawning lets each agent run its discovery + work phases without competing for the same window.

### Exceptions

- **2 bg agents parallel** is usually safe if both have small scopes (≤ ~50K tokens estimated each; ≤ 30 tool calls)
- **Foreground main-session work alongside 1 bg agent** is the canonical pattern — exploit it for productivity. Round 91 main session shipped 3 PRs (#315 / #316 / #318) in parallel with #468 running
- **Quick utility agents** (e.g., a focused grep across the portfolio that returns in 1-2 tool calls) can run in parallel without saturating

### Companion gotcha: shared working tree contention

Background agents and the main session **share the same working tree on the filesystem** — `git status` shows files staged by either side. If the main session runs `git add <specific-file> && git commit`, but another agent has staged different files concurrently, `git commit` includes ALL staged files. Observed Round 90 #466's AGENTS.md getting committed to #465's PR.

**Mitigation**: when running main-session work in parallel with a bg agent, use **branches** (not staged files on main) and explicit `git add <path>` per file. Don't trust `git status` cleanliness between bg-agent operations.

## R-PARALLEL-HUB-AGENTS — coordination protocol for concurrent hub sessions (2026-07-08)

**Multiple Claude Code sessions run against this hub (and the shared portfolio) at the same time. They share ONE git origin per repo, ONE Gemini API key, ONE R2 bucket, and ONE working tree per clone — so uncoordinated concurrent writes collide.** This section is the single consolidated protocol; the sibling sections it references (§ Stagger Background Agents / § Pre-work origin verification / § Verify origin state before claiming coverage / § Stale-clone recovery / `portfolio.md` § Per-repo pull-then-audit / `spark-anvil-website.md` § R-GEMINI-KEY-SERIAL) remain the detailed treatments — this rule unifies them and adds the cross-session coordination the individual rules didn't cover.

### Why this exists — collisions observed in a single session (all from concurrent hub agents)

1. **Queue-number collision** — a parallel session and this one BOTH added `## V35` to the work queue → git merge conflict; resolved by renumbering to `V36` and keeping BOTH entries.
2. **Rule-sync `set -e` abort** — `copy_rules_to_repos.sh` committed to a diverged app (a parallel session pushed between the sync's pull and push); the failing `git push` under `set -e` aborted the whole run, silently skipping ~54 apps (fixed reactively via resilient push, hub PR #1157 — but the ROOT is concurrent pushes).
3. **Per-app push race** — an app needed a rebase onto a parallel commit before its rule push landed.
4. **PR-merge races** — origin/main advanced mid-round (parallel merges), so hub PRs needed branch-update-then-merge.
5. **R2 bucket mutated mid-audit** — a backup audit's bucket count grew +20 objects (a parallel creative wave uploaded new audio + left `.vtt` uncommitted) between snapshot and hardening.
6. **Stale-clone / `index.lock` contention** — a backgrounded `git add` left a stale lock; a stale local `main` pointer.

### The protocol (8 disciplines)

1. **Queue-number allocation — never hard-code the next `V<N>`.** At the moment you commit a work-queue entry, `git pull` the queue first and take `max(existing V-numbers) + 1`. **Compute that max across the WHOLE file, not the recent top-of-file lane** — the web-clone lane's numbers (240s+) overlap the numeric range of OLD deep entries still in the file, so a "top-of-file max+1" pick silently collides with an old buried `## V<N>` header (2026-07-15, focusforge: picked V243 [top lane said 242] and hit BOTH a parallel-session V243 (mathlore) AND an old deep V244 (dealtales) → two rebase cycles; use `grep -oE '^## V[0-9]+' <queue> | grep -oE '[0-9]+' | sort -n | tail -1` and, if unsure, pick the first fully-free number above it: `grep -c "^## V<n>\b"` == 0). Expect + resolve queue merge-conflicts by **renumbering your entry and keeping BOTH** (the V35/V36 precedent) — never clobber the other session's entry. Same discipline for ADR numbers (`ADR-NNN`) and any other monotonic ID authored across sessions.

2. **Rule-sync single-flight.** Only ONE session runs `copy_rules_to_repos.sh --apply` (or any full portfolio distribution) at a time — concurrent runs race per-app pushes. Acquire the hub lockfile `.claude/.rule-sync.lock` (PID + ISO-timestamp + short purpose) before starting; remove it when done. If the lock exists and its PID is alive, WAIT or coordinate — do not start a second sync. A stale lock (dead PID, or timestamp > ~30 min old) may be reclaimed after verifying no sync is actually running (`ps` the PID). The resilient push (PR #1157) is the SAFETY NET, not a substitute for the lock.

   ```bash
   LOCK=.claude/.rule-sync.lock
   if [ -f "$LOCK" ] && kill -0 "$(cut -d' ' -f1 "$LOCK" 2>/dev/null)" 2>/dev/null; then
     echo "rule-sync in progress by PID $(cat "$LOCK") — WAIT/coordinate"; exit 1
   fi
   echo "$$ $(date -u +%FT%TZ) rule-sync" > "$LOCK"
   trap 'rm -f "$LOCK"' EXIT
   ```

3. **Bucket-/key-mutating single-flight.** Extend R-GEMINI-KEY-SERIAL's "one op at a time" to ANY shared-resource-mutating op: R2 uploads/prunes, the R2 backup sync, and any Gemini gen/gate/portrait run. An audit or backup must not race a live upload wave. After any wave that mutated R2, re-run the backstop auditor (`audit_r2_upload_coverage.py` / `audit_asset_backup_coverage.py --ci-mode`). If a bucket count shifts mid-audit, a parallel wave is writing — pause, let it finish, re-snapshot.

4. **PR-merge-race discipline.** Immediately before `gh pr merge`, `git pull --ff-only` your branch's base (or fetch + rebase the branch). If `gh pr merge` reports not-mergeable because origin/main advanced, **update-branch-then-retry** — do NOT `--admin`-force past a real divergence. `--admin` is only for genuinely non-blocking CI (Cloudflare build IN_PROGRESS, optional lints) per § Companion utility above, never to bypass a merge conflict a parallel session created.
   - **Merge-race treadmill escape (codified 2026-07-15 V255/V256).** During **peak parallel-session churn on the HUB repo**, `origin/main` can advance every few seconds — faster than GitHub recomputes mergeability after your force-push — so a PR touching the **shared hotspots** (`WORK_QUEUE_*` top-insert, `.claude/CLAIMS.md` EOF-append, `PORTFOLIO_PATTERNS.md`) can stay `CONFLICTING`/`UNKNOWN` through repeated rebases (a treadmill). Two reliable escapes: **(a) shrink the PR to non-hotspot files** — split the canonical doc/rule edits (which siblings rarely touch) onto their own branch; that PR stays mergeable through the churn, and the hotspot bookkeeping (work-queue/CLAIMS) rides a separate small PR or the next natural commit; **(b) tight rebase→force-push→poll-merge loop** — `git fetch origin main && git rebase origin/main && git push --force`, then poll `gh pr view <n> --json mergeable` until `MERGEABLE` and merge in the same window. Prefer (a) for an important/large PR (land the load-bearing content cleanly); use (b) for the small hotspot follow-up. **(c) CI-proven-green admin-merge in the post-rebase mergeable window (last resort for a hot-FILE treadmill vs a SLOW required-check).** When the conflict is a *content* hotspot (a union-merged file like `src/styles/play.css`) AND a required check is SLOW (the ~8-min Playwright e2e), the branch can re-conflict faster than each CI cycle finishes, so the loop never converges (2026-07-16 rupturerepair: ~5 rebase→8-min-CI→re-conflict loops on the `play.css` union tail). The sanctioned escape, ONLY when ALL hold: (1) the required checks have PASSED on equivalent trees ≥2–3× already (the code is proven green — the rebase delta is purely the mechanical union-seam brace-fix + folding in *disjoint* sibling files); (2) you RESOLVED the conflict via `git rebase origin/main` (so `mergeable=MERGEABLE`, not `CONFLICTING`); (3) you RE-RAN the local gates on the merged tree (`check-play-css-parse.mjs` + the touched clone's Vitest; `build:play` if time permits) green; then **`gh pr merge <n> --merge --admin`** in the tight window right after the force-push (poll `--json mergeable` → `MERGEABLE`, merge within ~1 min before the next sibling re-conflicts). This is **NOT** "--admin past a conflict" (the conflict is resolved) and **NOT** bypassing a check that might catch a real bug (it passed ≥2–3×) — it bypasses only the *redundant CI-re-run wait* on a mechanically-equivalent tree. Document the admin reason in the merge. Do NOT use (c) when the rebase changed anything non-mechanical, when CI hasn't proven the branch green, or to skip a first CI run. **Gotcha:** `gh pr checkout <n>` resets the index to the PR tip while leaving `HEAD` on your current branch — committing that state can **delete a sibling's files that landed on main after your branch point**; after a `gh pr checkout`, `git status` + `git reset --hard <known-good-ref>` before re-committing (cherry-pick your fix commit onto fresh `main` instead).
   - **(d) Sharded-e2e AGGREGATOR-LAG admin-merge — the aggregator required-check sits QUEUED behind a saturated runner pool while its shards are already SUCCESS (codified 2026-07-17, Math expansion wave).** The site e2e gate is an aggregator job (`R-WEB-CLONE-TEST-PERF`): the real work runs in `Vitest` + `e2e scope` + `e2e shard (N)`, and the REQUIRED check `Playwright (a11y + SEL-safety smoke)` is a trivial blob-merge job that runs AFTER the shards. When you fire many clone PRs in a **rapid cadence** (a whole cluster wave), the GitHub Actions **runner pool saturates** → shards go `QUEUED` for many minutes and the aggregator queues behind them, so `mergeStateStatus` stays `BLOCKED` long after the code is proven. The sanctioned escape, ONLY when ALL hold: (1) `Vitest` + `e2e scope` + **every** `e2e shard (N)` are `COMPLETED/SUCCESS`; (2) the ONLY non-green required check is the `Playwright (a11y + SEL-safety smoke)` aggregator, still `QUEUED`/`IN_PROGRESS`; (3) `mergeable=MERGEABLE` (no conflict — NOT a treadmill). Then **`gh pr merge <n> --merge --admin`** — this bypasses ONLY the redundant aggregator, which merges already-green shard blobs and **cannot fail** if the shards passed. It is NOT bypassing a check that could catch a bug (the shards ARE the tests; the aggregator just reports them). Do NOT admin-merge if any shard is still `QUEUED`/`IN_PROGRESS`/failed, or if `mergeable≠MERGEABLE`. **Prevention:** a rapid same-repo PR cadence self-saturates the runner pool — expect `QUEUED` waits and either space merges out or use this (d) escape once shards are green; poll-to-`CLEAN` first and only fall to (d) after confirming the shards-green + aggregator-only-laggard shape. (Distinct from (c): (c) is a hot-FILE conflict treadmill; (d) is a no-conflict runner-backlog aggregator lag.)
   - **(e) Runner-pool TRAFFIC JAM — the e2e SHARDS THEMSELVES sit `QUEUED` (not just the aggregator) for a long stretch: this is a WAIT, not a task, and it is RESOLVED by an autonomous poll-and-merge, NEVER by admin-merge (codified 2026-07-18, grade-3 clone ship #934).** When many PRs across the portfolio are in flight, the shared GitHub Actions runner pool saturates and a FULL-run PR's `e2e shard (1..4)` stay `QUEUED` (status `Q`, not `IN_PROGRESS`) for 15–40+ min while `Vitest` + `e2e scope` may already be `SUCCESS`. **The §(d) admin-escape does NOT apply here** — (d) requires **every shard `COMPLETED/SUCCESS`** with only the aggregator lagging; when the shards are still `QUEUED` they have not run, so admin-merging would bypass the actual tests (a real gate). **The correct resolution is to WAIT for the runners to free up — the jam self-clears — and make the merge autonomous so you don't babysit it:** launch a **background poll-and-merge loop** that polls `gh pr view <n> --json mergeStateStatus` every ~90–110s and runs `gh pr merge <n> --merge --delete-branch` the instant it reads `CLEAN` (guard: also break + surface if any check flips to a real `COMPLETED/FAILURE`). This lands the merge the moment CI clears without holding the foreground. A FULL run is proven-green when your LOCAL gates already passed (build:play + the relevant Vitest + screenshot-DoD) — so the wait is purely infrastructural, and the shards WILL pass when they run. Reference: #934 sat with all 4 shards `QUEUED` ~30 min behind the pool; the bg loop merged it at `CLEAN` with every shard `SUCCESS` (the local screenshot-DoD had already proven zero console errors, which is exactly what the smoke/a11y/interaction/keyboard shards assert). **Two harness gotchas for the poll loop:** a long `sleep` chain auto-backgrounds past the ~2-min foreground cap (expected — let it background + notify); and `--delete-branch` will fail to delete the LOCAL branch while a worktree is still checked out on it (harmless — the *remote* branch is deleted; remove the worktree at close-out, § R-SITE-WORKTREE). **PREVENTION (the real fix — stop causing the jam):** (1) a rapid same-repo PR cadence self-saturates the pool — **space merges out** across a cluster wave rather than firing all at once; (2) a **FULL run is far heavier than a scoped run** (4 shards over all clones vs ~48 tests for one clone) — a FULL run is triggered by touching a SHARED surface (`_shared/`, `play.css`, `pages/play/index.astro`, `playwright.config.ts`, …), so **when possible split the shared-surface edit onto its own small PR and keep the per-clone PR scoped** (§(a) treadmill-escape (a) is the sibling of this), which both merges faster AND consumes far fewer runners. (Distinct from (d): (d) = shards GREEN + aggregator queued → admin-OK; (e) = shards still QUEUED → wait + autonomous-merge, admin-FORBIDDEN.)
     - **(e-diag) A PR that has been `BLOCKED` for HOURS is almost always REPEATED FULL-RUN re-triggers + runner contention — NOT one hung run. Check the CURRENT run's per-JOB start times before you rerun/cancel; do NOT reset a healthy-but-slow run mistaking the PR's open-AGE for a stuck job (2026-07-20 audit, #1085).** A PR's *open age* (`createdAt` → now) and a *run's* age are different clocks: each branch re-push (rebase, fix, sibling-merge-induced retrigger) starts a FRESH `play-tests` run, so a PR can sit `BLOCKED` 3 h across several superseded runs while its CURRENT run is only ~10–12 min in. Before treating it as hung: `gh run list --commit $(gh pr view <n> --json headRefOid -q .headRefOid) --json databaseId,status,createdAt` → the newest run; then `gh run view <id> --json jobs -q '.jobs[]|"\(.status) \(.name) started=\(.startedAt)"'`. If the shards' `startedAt` is RECENT (minutes ago) and `IN_PROGRESS`, it's healthy-but-slow (a FULL run + pool contention, §(e)) → **WAIT / bg poll-merge, do NOT `gh run rerun`** (rerun RESETS the ~12-min progress → strictly worse) and do NOT admin-merge (shards haven't finished → real gate). Only a run whose jobs started long ago (≫ the normal ~3–4 min/shard) with no progress is genuinely hung → then rerun/cancel. And it's another lane's PR to *merge* — investigate/unblock is a courtesy; the owning lane merges it (§5, territory). This is the run-level sibling of § R-BUILD-FAILURE-READ-FULL-LOG-FIRST (read the actual run state, don't infer from the PR's age).
     - **(e-diag2) A PR-scoped smoke that touches a SHARED SURFACE (`play.css`/`_shared/`/`index.astro`/`playwright.config`) runs FULL-scope on a SINGLE runner — so it can take 40+ min and LOOK stuck while being slow-BY-DESIGN, NOT hung; distinguish by reading the JOB STEP, never the elapsed time (2026-07-21, agoraquest #1158 ~42-min smoke).** The post-merge `play-tests-full` is 6-way SHARDED (fast); the per-PR `play-tests` gate is NOT — its Playwright job is a SINGLE runner, and when the PR touches a shared surface the "PR-scoped, round-play-excluded" smoke expands to the ENTIRE `/play` surface (all clones × light+dark) on that one runner. Under a saturated Ubicloud ARM pool that is legitimately 40+ min. **The diagnostic is the running STEP, not the clock:** `gh run view <id> --json jobs` → if the job's `in_progress` step is `Playwright (…)` (actively executing the test step, setup steps done) it is HEALTHY-but-slow → WAIT (or the (e-diag3) proven-green escape); if it is wedged on `npm ci`/`actions/cache` for many minutes, or the job is QUEUED (no runner assigned), it is contention/genuinely-stuck → investigate. A reported "stuck runner" is very often this slow-single-runner shared-surface smoke; and the orphan PR-run **auto-cancels** the instant the PR merges + branch is deleted (not an orphan to chase). **Cost corollary (R-COMBINE / R-WEB-CLONE-TEST-PERF):** because a shared-surface touch turns even the PR gate into a full-scope single-runner run, SPLIT the shared-surface edit onto its own PR and keep per-clone PRs scoped — a scoped clone PR's smoke is minutes, not 40+.
     - **(e-diag3) The proven-green escape for (e-diag2): when a shared-surface PR's slow single-runner smoke is the sole blocker, §4(c) admin-merge is warranted — and `build:play` exit-0 + link-check-OK is the strongest LOCAL proxy for the deploy gate (2026-07-21, agoraquest #1158).** The single-runner full-scope smoke asserts render + zero-console-error + a11y across `/play`; you can independently PROVE that green locally: the PR-CI **Vitest** check SUCCESS + the clone's local **Vitest** + **`node scripts/check-play-css-parse.mjs`** + **`npm run build:play` exit-0 with the link-check OK** (the actual deploy-breaker gate — a broken import/route 404s here) + the **dual-scheme screenshot-DoD** + (where a purely-mechanical rebase followed an already-all-green CI run on byte-identical content) that prior run. With ALL of those green and `mergeable=MERGEABLE` (conflict RESOLVED via rebase, not bypassed), the pending smoke is a redundant re-run on a proven tree → `gh pr merge --merge --admin`, documenting the basis. Do NOT use this to skip a FIRST CI run, or to merge past an unresolved conflict.

5. **Territory claiming (lightweight).** A session doing sustained work in a specific app repo, or on a portfolio-scale wave (e.g., the multi-beat chapter sweep, a per-cluster gen wave), ANNOUNCES it — a line in the session handoff doc AND/OR a `.claude/CLAIMS.md` entry (`<repo-or-wave> · <session/PID> · <ISO-timestamp> · <purpose>`). Other sessions read claims before starting concurrent writes to a claimed repo/wave and pick different territory. The "V35 owned by parallel session" note is the informal precedent this formalizes. Claims are advisory, not locks — they prevent the expensive collisions (duplicate paid gen, working-tree churn), not every edit. **A claim is a coordination signal ONLY once it is PUSHED to origin — a claim committed (or merely edited) LOCALLY is invisible to every sibling session, so it protects nothing.** In a hot fleet (many concurrent clone/content lanes), **push the `.claude/CLAIMS.md` entry to origin BEFORE sustained authoring, not at PR time** (`.claude/CLAIMS.md` is `merge=union`, so a one-line claim commit fast-forwards cheaply + conflict-free — pull-rebase-push, retry on the merge-race). Reference: 2026-07-16 taletrail — the lane was claimed locally but left unpushed while the build proceeded; a sibling picking the next younger-cluster ELA clone (tinyletters was already in flight) could not have seen it until the claim was pushed. The same "pushed = visible" discipline applies to the RELEASED note at lane close.
   - **🔴 CRITICAL — CHECK for an existing claim by the CLONE SLUGS you intend to work — NOT by the cluster-name PHRASING — because a sibling may describe the SAME lane with different wording (2026-07-17, dual SEL-expansion collision).** A pre-claim CLAIMS grep for cluster phrasing (`'SEL lane'` / `'SOCIAL & EMOTIONAL'`) MISSED a sibling's already-active same-lane claim worded `SEL-web-clone-expansion (8: tempcheck·focusforge·…)`; the parallel session surfaced only later via a per-clone-slug grep + the §5a origin re-check — by which point both sessions had claimed the same 8-clone list. So a **cluster/lane-level claim is NOT exclusive** — two sessions CAN claim the same cluster near-simultaneously (even two of the same founder's parallel sessions on one program), and a phrasing-only search won't reveal it. Before claiming a lane: `for a in <slug1> <slug2> …; do echo "$a: $(grep -c "\b$a\b" .claude/CLAIMS.md)"; done` (per-slug), and if a sibling is already on the list, apply §5c (split from the opposite end) UP FRONT rather than discovering it mid-build. Same scan-correctness class as the `find`-not-`ls` / `${=VAR}`-word-split / grep-both-syntactic-forms traps: a search that under-matches yields a wrong "lane is free" conclusion. The per-clone origin re-verify (§5a/5b) is the real backstop — the claim is a signal, not a lock (§5d). **🔴 CRITICAL: every claim MUST lead with a machine-readable `slugs=<comma-separated-slugs>` token** (codified in the `.claude/CLAIMS.md` header, 2026-07-17) listing EVERY app/clone it covers — a claim without the token is a defect, because it re-opens the prose-grep gap that let two sessions claim the same 8-clone list. With the token the pre-claim check is a reliable `grep -oE 'slugs=[a-z0-9,-]+' .claude/CLAIMS.md | grep -w <slug>` instead of a fragile prose grep; pre-token (pre-2026-07-17) entries fall back to the per-slug word-grep + the origin-file check.
   - **🔴 CRITICAL — the pre-claim CLAIMS check MUST scan the WHOLE file (`grep` for your slugs/cluster), NEVER `tail -N` / `head -N`; AND a `PAUSED … lane CONTINUING / N remain / resume via <handoff>` entry is a HELD lane, not an available one (2026-07-18, Science-cluster collision).** The CLAIMS file grows to hundreds of lines in a hot fleet, so reading it with `tail -40` shows only the most-recent entries and **silently truncates every claim above the window**. Reference incident: a session read CLAIMS with `tail -40` (lines 767–807 of an 807-line file), concluded "Science is the one large unclaimed cluster," claimed 30 Science clones, and **shipped Wave-1 + built Wave-2** — before discovering a `PAUSED … Science lane CONTINUING, 30 science clones REMAIN, resume via <handoff>` entry **at line 761** (6 lines above the `tail -40` window) plus a live back-half splitter; the Wave-2 build then collided with the resuming lane + a richer sibling and had to be **yielded (site PR closed, hub rows reverted)** — a whole wasted build the full-file grep would have prevented. **The rules:** (a) `grep -niE '<slug1>|<slug2>|…|<cluster>' .claude/CLAIMS.md` over the ENTIRE file (never a tail/head slice) before claiming; (b) treat a **`PAUSED`/`WIP`/`lane CONTINUING`/`N remain`/`resume via`** entry as **owned, not released** — only an explicit `RELEASED`/`✅ COMPLETE`/`lane released` closes a lane (a paused lane's owner will resume, and other sessions may already be splitting its remainder per §5c). When the target lane is paused-but-continuing, pick genuinely different territory or coordinate a §5c disjoint split UP FRONT — do not read "paused" as "free."
   - **⚠ YIELD the whole lane (don't scope-reduce-and-re-race) when a sibling is SYSTEMATICALLY sweeping it with a strict-SUPERSET deliverable AND is faster (2026-07-18, Science Wave-2).** §5b says "sibling shipped a clone first → discard your version of THAT clone." This sharpens it for the *whole-lane* case: when the sibling is churning through the cluster in batches (`expand(<cluster> batch 1/2/3)`) with a deliverable that is a **superset** of yours (they ship pass-and-play + room MP **+ adventure**; you ship only pass-and-play + room MP) and landing faster than you can rebuild-around-the-collision, the treadmill is unwinnable and your output is redundant even where it lands — so **yield the entire remaining lane** (close your PR, revert the now-invalid hub rows), rather than re-racing a thinner subset each cycle. KEEP any COMPLEMENTARY sub-wave you shipped that the sibling is NOT doing (the Science rails+mastery Wave-1 stood because the sibling only did shells+adventure). The yield is the disciplined outcome, not a failure — it stops burning the saturated runner pool on duplicate work.

   **5a. A handoff's "NEXT PRIORITIES" list is a DOUBLE-PICK MAGNET — claim-before-authoring + check-for-the-deliverable-before-authoring (2026-07-14 V224 collision).** The exact thing that makes a good handoff — a crisp, ordered "⭐ NEXT PRIORITIES" list — is what makes two independently-resuming sessions pick the *same* top item and duplicate it. So before you START a discrete deliverable named in a handoff/queue (a research doc, a rule, an ADR, an audit, a single-app port), you MUST, in this order: (a) **`git fetch origin main` and check the deliverable does not already exist** — grep origin for the target FILE (`git show origin/main:Docs/<THE_DOC>.md`), the rule token (`git show origin/main:.claude/rules/<f>.md | grep <RULE-ID>`), the queue line's status flag, AND `gh pr list --search "<topic>"` (open + recently-merged, ALL authors — every hub session pushes as the same GitHub user, so `--author @me` ≠ "only me"). **⚠ A directory/tree existence check MUST test the OUTPUT, not the exit code — `git ls-tree origin/main -- <path>` returns exit 0 whether or not `<path>` exists, so `git ls-tree … && echo EXISTS` ALWAYS fires a false positive** (2026-07-16 creaturecare: a `&& echo "EXISTS ON ORIGIN — STOP"` fired on an empty result and briefly read a clean, un-built clone as already-shipped). Count the lines: `[ "$(git ls-tree -r origin/main --name-only -- <path> | wc -l)" -gt 0 ]`, or use `git cat-file -e origin/main:<file>` for a single file (which DOES set exit code — the check 5b already uses). Same scan-correctness class as the `find`-not-`ls` + `${=VAR}`-word-split traps. (b) **append a CLAIMS line for that specific item FIRST** (before authoring), not bundled into the shipping PR — an unclaimed item authored over 20 min is 20 min a sibling can duplicate it; (c) if it already exists / is claimed, **STOP and treat it as done-by-sibling** — verify their version is complete, delete your local duplicate branch, and move on (do NOT open a second PR). Reference incident: two sessions both resumed the same handoff, both authored `RESEARCH_YOUNGER_CLUSTER_CAST_SIZE_2026-07-14.md` + `R-YOUNGER-CLUSTER-CAST-SIZE` on the same branch name; caught at rebase (conflict), the sibling's PR #1599 had already merged, the duplicate was abandoned with zero shipped waste — but only because the pre-push rebase surfaced it. Doing (a)+(b) up front avoids the wasted authoring entirely. This is the item-granular sharpening of discipline 1 (queue/ADR numbers), discipline 7 (verify-origin), and `workflow.md` § Cross-Repo Audit Methodology Rule 4 (verify-before-action).

   **5b. A LONG authoring/build window needs a SECOND origin re-check right before you SHIP — the pre-start check goes stale (2026-07-14 rupturerepair collision).** Rule 5a's check-before-authoring is necessary but NOT sufficient when the work itself is long (hand-authoring a 16×25=400-item kit ≈ 30–60 min; building a `/play` clone; a big doc). A sibling can pick the same item and ship it *during* your authoring window, so an "uncontested" reading at START is worthless by the time you finish. Therefore, **immediately before the first shipping commit (the app-repo commit / the PR), RE-RUN the origin existence check** — `git fetch origin main` + `git cat-file -e origin/main:<the target file>` for BOTH the deliverable (e.g. `<app>-app/Resources/.../kit_01*.json`, the hub porter `scripts/port_<app>_*.py`) AND its handoff/registry marker — and if the sibling already shipped it, **DISCARD your local work uncommitted (`git checkout -- <paths>` / `git clean -fd`), do NOT open a PR, and move to the next uncontested item.** Nothing ships duplicated; only the authoring time is lost (which 5c below reduces). Two natural tripwires that surface the collision for free — heed them: (i) a `Write` to the handoff doc **failing because the file already exists** (the sibling wrote it) is a positive collision signal — Read it + check origin before proceeding; (ii) a pre-push `git rebase`/`git pull` **conflict on a shared file** (5a's V224 tripwire). Reference incident: `rupturerepair` read "uncontested" at authoring start, but the sibling lane merged its 16×25 kit PR + hub porter during the ~30-min authoring window; caught when the handoff `Write` failed (file already on origin), the local duplicate was discarded uncommitted, zero shipped waste. **5c — shrink the window / pick disjoint items to make collisions rare:** when a sibling is demonstrably racing the SAME ordered work-list (you keep finding items freshly shipped), don't march the SAME order — **pull the list from the OPPOSITE end, or a different cluster**, so you and the sibling cover disjoint items instead of colliding on the same next one (the board-game cohort shipped collision-free precisely because it was a disjoint sub-list); and prefer author-then-*immediately*-ship over batching several apps before pushing, so each item's exposure window is minutes, not an hour.

   **5d. "Claim FIRST" means PUSH the claim to origin FIRST — a local-only CLAIMS append coordinates NOTHING (2026-07-16 taletrail collision).** 5a's "append a CLAIMS line FIRST" is necessary but INSUFFICIENT if the append stays local: a CLAIMS line that is never committed + pushed is invisible to every sibling, and you stay blind to theirs until your next `git fetch`. The claim step is therefore ATOMIC and comes BEFORE the worktree / scaffold / any authoring: **`git fetch origin main` → append your one-line CLAIMS entry → immediately `git commit .claude/CLAIMS.md` + `git push` to origin.** It is a tiny disjoint append to the union-merged `.claude/CLAIMS.md`, so it lands in ~5s and effectively never conflicts; the `git pull --ff-only` you do before pushing is what surfaces a sibling's competing claim — if one fast-forwards in, you've caught the collision in SECONDS and yield before spending any build time. Only after the claim is on origin do you create the worktree + build (then re-fetch at heavy-authoring start + before ship, per 5a/5b). **Reference incident: taletrail** — a session appended its claim LOCALLY and built the entire younger-cluster clone (content model + character-builder + story-maker + 5 pages + CSS, ~20 min) while a sibling independently claimed taletrail on origin and pushed; the local session was invisible to the sibling and blind to the sibling's claim until a manual `git fetch` (founder-prompted), and avoided shipping a duplicate ONLY because nothing had been pushed — the yield cost zero *shipped* work, but ~20 min of authoring was wasted, which pushing-the-claim-first would have prevented outright. The rule of thumb: **the claim is not a note-to-self, it is the coordination signal — it does not exist until origin has it.**

   **5e. A resume-HANDOFF's backlog is a SNAPSHOT — a live sibling may have SUB-CLAIMED part of your OWN lane since it was written, so derive your real scope as (handoff backlog) MINUS (every sibling's active `slugs=`), NOT the handoff's list/count (2026-07-18 SCIENCE-cluster front/back split).** When you resume a large-cluster lane from a `CONTEXT_HANDOFF_*` that lists "N remaining," that list was true at write-time; but on a big cluster the founder explicitly runs MULTIPLE hub agents (§ opening + the founder's own "more than 1 hub agent can work on it"), and a sibling may have appended a `CLAIM: slugs=… · pid-<lane>-BACK-cN` carving off part of the SAME backlog AFTER the handoff was authored. Working the handoff's ordered list top-down then collides with the sibling on the overlap (typically the tail). So the FIRST step of resuming is NOT "start item 1" — it is: (a) `git fetch origin main` + grep EVERY active (`CLAIM`/`UPDATE`, not `RELEASED`) `slugs=` token for your cluster (`grep -E '^(CLAIM|UPDATE):' .claude/CLAIMS.md | grep -oE 'slugs=[a-z0-9,._-]+'`); (b) compute YOUR scope = handoff-backlog **minus** every sibling's active slugs; (c) if a sibling took the BACK, you take the FRONT (the §5c split-from-opposite-ends, now applied to a *resumed* backlog, not a fresh one); (d) push a NARROWED `slugs=` claim for exactly your complement (do NOT re-assert the whole handoff list — that re-overlaps the sibling) and cross-check each remaining item against origin ground truth (already-shipped/expanded → drop it, per 5a). Reference: resuming the science handoff (30 "remaining"), a live sibling `pid-wc-science-BACK-c2` had sub-claimed 15 from the tail since the handoff was written; the correct scope was the disjoint FRONT 15, derived by set-difference on the `slugs=` tokens — not the handoff's 30. This is §5a ("a NEXT-PRIORITIES list is a double-pick magnet") generalized from a single next-item to a whole resumed backlog under an active co-worker: **trust the live `slugs=` tokens over the handoff's snapshot count.**

   **5f. Two lanes claiming the SAME `slugs=` for DIFFERENT AXES both pass the slug-overlap check yet COLLIDE on the shared per-clone artifacts — the claim token must encode the AXIS/scope, not just the clones (2026-07-18 Math axis-7 vs axis-8 collision).** The §5/§5a–§5e discipline detects overlap by comparing `slugs=` sets — but a claim's `slugs=` says WHICH clones, not WHICH axis/pass. So when lane A does **axis-7 (`/review`)** and lane B does **axis-8 (`<MasteryProgress>`)** on the SAME clones, a `slugs=`-only check reads them as "different work on overlapping clones" (or misses the overlap entirely if the phrasings differ), both proceed, and they collide anyway on the SHARED per-clone files every expansion touches: the landing `index.astro` (both add a card/link), the clone's `PARITY_WEB_VS_IOS.md` (both add a `### Pass N` row), and — worst — the **same-dated `AUDIT_WEB_CLONE_EXPANSION_<app>_<date>.md`** filename (a git `AA` both-added conflict). Reference: my Math `pid-wc-math-axis78-opus` lane (axis-7+8) collided with a sibling `mathlogiccs-axis8` lane (axis-8) on ratiorealm/equationquest/functionforge/geometryforge — the sibling's axis-8 hub docs (V364) merged mid-build, my hub-docset rebase hit 5 conflicts, and the site `index.astro` would have double-added `<MasteryProgress>`. **Prevention + resolution:** (a) a claim `slugs=` line SHOULD name the axis/scope (`… axis=7` / `… pass=2-axis8`) so a sibling sees the axis, not just the clones; (b) at pre-claim/pre-ship, when the same clones are already claimed by another lane, check WHICH AXIS the sibling is doing (their PR title / hub `AUDIT` / PARITY row), and if different, **split by axis** — cede the sibling's axis, ship only your disjoint axis; (c) make the artifacts non-colliding: **AXIS-suffix the AUDIT filename** (`AUDIT_WEB_CLONE_EXPANSION_<app>_AXIS7_<date>.md`), add a distinct PARITY sub-row (`### Axis 7 …`, not a duplicate `### Pass 2`), and place the landing edit at a **different anchor** than the sibling's (a Mixed-practice link near the practice card vs. a MasteryProgress card after the hero → clean auto-merge, no double-render). This is the axis-granular form of §5a/§5e: slug-overlap is necessary-but-insufficient; the real disjointness test is (slugs × axis), and the shared per-clone files (`index.astro`/`PARITY`/dated-`AUDIT`) are the collision surface to keep disjoint.

   **5g. A RELEASED lane STILL shows its open CLAIM/UPDATE/EXTEND lines in the append-only union-merged log — decide "is this lane live?" by PID (CLAIM↔RELEASED exact match), NEVER by `slugs=` alone; and a RELEASED pid that differs from its CLAIM pid by a TYPO (an apostrophe / one-character variant) splits the pairing so a genuinely-DONE lane reads as still-open (2026-07-18 Science stale-claim reconciliation).** `.claude/CLAIMS.md` is append-only + `merge=union` (§ R-WEB-CLONE-MERGE-HYGIENE Fix 1), so releasing a lane APPENDS a `RELEASED:` line — it does NOT remove that lane's earlier `CLAIM`/`UPDATE`/`EXTEND` lines. So the standard §5d/§5e `slugs=` grep (the pre-claim / pre-resume / "is this cluster free?" check) WILL hit the stale-but-released open lines, and read naively it misreads a FINISHED lane as active — the exact confusion that made the whole Science cluster look double-claimed when all 5 of its lanes were actually done. Worse, the live-vs-done pairing is matched on PID, and a PID typo breaks it: the Science FRONT lane's CLAIM used `pid-wc-science-FRONT-73's` (apostrophe) while its RELEASED used `pid-wc-science-FRONT-73s` (no apostrophe), so a pid-exact audit saw the CLAIM as unreleased. **The discipline:** (1) to decide whether a lane is live, match its PID across CLAIM↔RELEASED — an open line whose pid has a corresponding `RELEASED:` (or `✅ COMPLETE` / `lane released`, per §5e) is DONE and its slugs are STALE; the slugs alone never tell you. (2) At lane close, append the `RELEASED:` with the **byte-identical pid** from the CLAIM (no typo, no reworded/re-cased pid) so the pairing is unambiguous — a mangled release pid re-opens this trap. (3) When stale-but-released open lines are emitting a false-active signal (a founder or sibling can't tell the lane is finished), append **ONE authoritative `CORRECTION:` entry** — union-safe + additive — carrying a `slugs=` token + naming each stale pid as RELEASED + reconciling any pid typo + naming the ONLY still-active lane; **NEVER edit or delete another lane's existing lines** (union-merge keeps both copies → semantic dup, and editing others' entries violates § discipline 5). This is the CLAIMS-hygiene sibling of §5f: §5f keeps two LIVE lanes disjoint; §5g keeps a DONE lane from masquerading as live. Reference: the 2026-07-18 Science cluster — all 5 expansion lanes (`pass1-opus48`, `BACK-c2`, `FRONT-73's`≡`FRONT-73s`, `axis7-opus48`, `orphans-opus48`) were RELEASED, but their open CLAIM/UPDATE/EXTEND lines + the FRONT pid apostrophe-typo read as active to a `slugs=` grep; corrected via one appended `CORRECTION (stale-claim reconciliation)` entry naming all 5 released pids + the sole active lane (`axis78-gaps-opus48`), founder-directed. **(4) The CORRECTION sweep is a PERIODIC maintenance action, not only an incident/founder-driven one — the append-only log accrues ~15 stale-but-released lanes over ~3 days of fleet activity (2026-07-20 audit), so a bottleneck/coordination audit (or any session that finds the "active" count implausibly high) is a natural trigger to append a fresh authoritative snapshot.** Keep it CONSERVATIVE to avoid false-closing a live lane: hard-assert RELEASED only for pids with explicit release-prose in their OWN lines (or a verified supersession); name the VERIFIED-ACTIVE lanes by an open-PR/paused-continuing check; and bucket the genuinely-ambiguous aged lanes (no open PR, no recent UPDATE) as "UNCERTAIN — verify origin + own-PR-list before claiming their slugs (§5a/§5b), owner re-assert if resuming" rather than asserting either way. Reference: `Docs/AUDIT_PIPELINE_BOTTLENECKS_2026-07-20.md` §4 → the 2026-07-20 `CORRECTION (pid-bottleneck-audit-opus48)` snapshot (4 verified-active / ~15 reconciled-released / ~16 uncertain).

   **5h. A held/PAUSED LANE claim does NOT reserve the lane's individual OPEN pickup items — a resumed lane's top open item is still a §5a double-pick magnet, and a sibling may ship it WITHOUT ever posting a CLAIMS line, so ONLY the origin-FILE check (not the `slugs=` grep) catches them; run the §5a per-ITEM origin check at authoring-START, not just at ship-time §5b (2026-07-21 ProtoForge double-authoring).** A lane-level claim (`slugs=<N apps> · pid-…`, especially one marked `PAUSED-CONTINUING`) reads to a sibling as *"the holder isn't actively working this right now, so its open pickup items are fair game"* — it advertises territory, it does not LOCK each item. So resuming your OWN paused lane and starting its top open item (the V455/handoff "P1") on the strength of the lane claim alone is unsafe: (1) the item is still the §5a NEXT-PRIORITIES double-pick magnet; (2) a sibling racing the same obvious P1 may be **mid-flight with NO CLAIMS entry** (authored + shipped without claiming), so the §5d/§5e `slugs=` grep shows the lane "uncontested" while the deliverable is already merging — **only the §5a origin-FILE existence check (`git cat-file -e origin/main:<the deliverable file>` + `gh pr list --search "<topic>" --state all`) detects a claimless sibling**; (3) §5b's ship-time re-check DOES catch it (the merge/PR-conflict tripwire fires exactly as designed — that is how this incident surfaced), but only AFTER the authoring is spent. **The discipline (resuming a held/paused lane):** before authoring each item, (a) run the §5a per-ITEM origin-FILE check even though you hold the lane; (b) push a NARROWED per-ITEM claim (`slugs=<app> · axis=<the-specific-deliverable>`), not just the coarse lane RESUME — an item-granular claim is what a sibling's §5a/§5e check can actually see; (c) if the file is already on origin, DISCARD your local work uncommitted, close any PR you opened, verify the sibling's version is complete, and move to the next uncontested item (nothing ships twice — only authoring time is lost). Reference incident: I held `pid-v426-1518steam-complete-opus48` (PAUSED-CONTINUING, 5 STEAM slugs) and pushed a lane-level RESUME claim, then hand-authored the full 400-item ProtoForge bank (~30 min) on the strength of that claim WITHOUT a §5a per-item origin check at authoring-start; a sibling had independently picked the same V455-P1, authored + merged `protoforge-app` PR #1 + the hub porter (with NO protoforge CLAIMS line to grep) during my window; my ship-time commit conflicted (§5b tripwire), I closed my redundant PR #2, discarded my duplicate porter, verified the sibling's bank (400 items / 400 unique ids / 0 errors), and moved on — zero duplicate shipped, but the authoring was wasted. This is the item-vs-lane sharpening of §5a (per-item check + per-item claim) and the claimless-sibling sharpening of §5e (the `slugs=` grep is necessary-but-insufficient; the origin-FILE check is the real backstop).

   **5i. TAKING OVER a KILLED agent's lane — adopt the pid, AUDIT the partial state against ORIGIN (never trust the claim's last "DONE" prose), finish-or-discard per §5b, release under the same pid (2026-07-21).** When a hub agent is KILLED mid-work (crash / founder stop / harness death), its CLAIMS lane stays OPEN (never `RELEASED`) with PARTIAL on-disk state — uncommitted files in the shared clone, staged-but-uncommitted hub artifacts, and deliverables that LOOK present but are INCOMPLETE (a "16 kits" bank that is actually 15/25; a porter whose fill-wiring line was added AFTER its last emit so the emitted output lags the source). This is DISTINCT from §5g (a RELEASED lane's stale-but-open lines) and §5h (a LIVE `PAUSED-CONTINUING` lane): here the lane is genuinely owned + open, the owner is simply DEAD — so a resuming session (or the founder reassigning it) TAKES IT OVER rather than treating it as free territory (would double-build) or waiting on it (nobody's coming). **The discipline:** (a) **confirm the owner is actually dead** — founder says killed, or the owning process/session is gone — NOT a live paused lane (§5h) you'd be stepping on; (b) **append an `UPDATE:` under the SAME pid** (byte-identical, so the lane's history stays one coherent thread — never mint a new pid) noting "TAKEOVER — prior owner killed" + your audit summary, and PUSH it first (§5d); (c) **AUDIT every deliverable against ORIGIN GROUND TRUTH, per-item — never trust the claim's "DONE this session" prose** (a killed agent's last checkpoint LAGS its death point, so the prose over-claims what actually landed): for each item check what is on origin (`git ls-tree`/`git cat-file -e origin/main:<file>`), what is local-only/staged/uncommitted, and what is present-but-INCOMPLETE (**validate the CONTENT, not just the file count** — the 15/25-kit trap: 16 files present ≠ 16×25 done); (d) **finish-or-discard each partial per §5b** (complete + ship it, or discard uncommitted if a sibling already shipped it); (e) **RELEASE under the same pid** at lane close (§5g). Reference incident (2026-07-21): took over `pid-adr059-academic-p1-opus48` (ADR-059 6-app 15–18 academic P1) after the founder said the owner was killed; the claim prose implied Phase-A kits were done, but the per-item origin audit found 5 apps genuinely complete (16×25 on origin + iOS handoffs) while **locusforge** had 16 kits at only **15/25** (the porter's `FILL`-merge line was added after the last emit → never re-run), uncommitted in the app repo, hub porters+research staged-uncommitted, and NO iOS handoff — none of which the claim prose disclosed. This is the killed-owner sibling of §5g/§5h: same "the claim is a signal, not ground truth" root (R-CANONICAL-DOC-GROUND-TRUTH), applied to a dead lane's partial work; the file-count-≠-content-complete trap is the §5b origin re-check applied to a resumed deliverable.

6. **Distribution scripts must be concurrency-safe.** Any script that writes to multiple repos MUST: (a) `git pull` per repo immediately before its push; (b) be **resilient** — never `set -e`-abort the whole batch on one repo's push failure; on failure `git pull --rebase` + retry, else record `PUSH-FAILED`, leave the commit local, and CONTINUE the loop (the V32 P2 resilient-push template); (c) be **idempotent / re-runnable** — a second run over already-synced repos is a no-op (merge-aware sync, skip-if-present gen); (d) **commit only its own pathspec, NEVER the whole index** — use `git commit <path> -m …` (pathspec commit), not `git add <path> && git commit` (which commits the ENTIRE index). A parallel session — or this environment's auto-staging of Write-created files — may leave UNRELATED files staged in the target repo's index; a whole-index commit sweeps them into the distribution commit. (2026-07-08 incident: a straggler `copy_rules_to_repos.sh` run swept another workflow's auto-staged untracked `HANDOFF_*` doc into an app's rule-sync commit because the commit wasn't pathspec-scoped. Fixed: `git commit .claude/rules/ -m …`.) **⚠ `git commit <path>` (pathspec commit) ERRORS on a NEW UNTRACKED file** (`pathspec '<path>' did not match any file(s) known to git`) — a pathspec commit only acts on TRACKED/STAGED paths. So for a brand-new file (a freshly-`Write`n handoff/doc), you MUST `git add <specific-path>` FIRST (a SCOPED add — never `git add -A`/`.` — is itself pathspec-safe: it stages only that path), then `git commit <same-path> -m …` (the commit-pathspec double-guards against any other file a sibling/auto-stage left in the index). Bit the 2026-07-18 VA&D expansion (10 app backport handoffs: `git commit <handoff>` no-op-errored on the untracked file → empty branch → PR "No commits between main and …"; fixed by `git add <handoff>` first). Reserve a bare `git commit <path>` (no prior add) for MODIFYING an already-tracked file. Never source the target list from a filesystem glob — use the canonical registry per `portfolio.md` § Distribution scripts MUST source from canonical portfolio registry.
   - **⚠ The whole-index-`git add` sweep can commit a `node_modules` SYMLINK into `origin/main` — and `.gitignore`'s `node_modules/` (dir rule) does NOT ignore the symlink (a FILE), so it slips through and BREAKS the Cloudflare deploy (2026-07-19, site commit a825af56).** The canonical hub/site worktree setup symlinks `node_modules` → the shared clone's `node_modules` (R-SITE-WORKTREE). That symlink is an **untracked file** in the worktree; a `git add -A`/`git add .`/whole-index commit sweeps it in — AND it dodges `.gitignore` because the ignore entry is `node_modules/` (trailing slash = *directory named node_modules*), which does NOT match a *symlink file* named `node_modules`. The committed result is a `120000` (symlink) blob pointing at an **absolute local path** that doesn't exist on Cloudflare's fresh build clone → a dangling symlink at `node_modules` that breaks `npm ci` / the play-unit build, so **every play-unit deploy silently stops updating** (all `/play/*` stay stale/404) with no CI signal (CI doesn't `git clone`-then-`npm ci` from scratch the way the deploy does). **Detection:** `git ls-tree origin/main node_modules` returns a `120000` blob (a tracked symlink) — it should return nothing. **Fix:** `git rm --cached node_modules` + add a **bare `node_modules`** line to `.gitignore` (matches both the dir and the symlink file), one tiny PR (reference: site #1076). **Prevention = discipline 6(d):** pathspec-`git add <your-files>`, never `git add -A`/`.` in a worktree that has a `node_modules` symlink. **Rebase gotcha:** a branch pushed BEFORE the fix still tracks the symlink, so `rm node_modules` (to swap in your working symlink) leaves an *unstaged deletion of a tracked file* that blocks `git rebase` ("cannot rebase: You have unstaged changes" / "untracked working tree files would be overwritten") — `git checkout -- node_modules` to restore it, THEN rebase (the rebase onto the post-fix main removes it cleanly via 3-way merge), then re-create your working symlink for tooling.

7. **Pull-first / verify-origin everywhere (the umbrella).** Every freshness query, every pre-work step, every coverage claim reads/writes against ORIGIN, not stale local state — per the sibling sections. In a parallel-session world these are not optional politeness; they are the only thing that keeps two sessions from acting on each other's stale view.

8. **Subagent-scope discipline — a delegated subagent stays in its lane.** A subagent (or background Agent) MUST be scoped to **disjoint, explicitly-named files** and MUST NOT (a) edit SHARED hotspots the main agent is concurrently working — `.claude/rules/*`, the work queue, `.claude/CLAIMS.md`, canonical process/template docs, shared code (`clones.ts`, `play.css`, `_shared/`) — or (b) INVENT scope: no new rules, ADRs, work-queue/V-numbers, or handoff docs the main agent didn't ask for. Delegate READ-heavy or app-namespaced work (a codebase deep-read, per-app doc stubs, a scoped implementation in one app's subtree); keep shared-file / rule / numbering / cross-cutting-doc edits in the MAIN agent, which serializes them. **Reference incident (2026-07-10):** a doc-writing subagent told to fill 4 app-namespaced stubs also hallucinated a "founder ask" and authored an unsanctioned rule + a competing work-queue entry + its own handoff in shared files — all reverted (nothing shipped). The failure mode is identical to two uncoordinated sessions (disciplines 1–7), one level down: an in-session subagent is a concurrent writer too. When a subagent returns, REVIEW its diff against the shared-hotspot list before staging.

9. **Parallel clone-expansion (or any parallel-subagent) HUB-SIDE commits need their OWN hub `git worktree` — the single shared hub clone's HEAD/index/branch is NOT concurrency-safe (2026-07-17).** When you fan out N clone-expansion subagents (`R-PARALLEL-WEB-CLONE-BUILD`), each already isolates its SITE work in a per-app site worktree (`R-SITE-WORKTREE`) — but a clone-expansion pass ALSO writes hub-side artifacts (`Docs/web/<app>/AUDIT_*` + `PARITY_WEB_VS_IOS.md`, a `.claude/CLAIMS.md` line, sometimes a pickup regen). If every subagent commits those in the ONE shared hub clone, they thrash a single working tree — `git` branch/HEAD/index/`ORIG_HEAD` are **per-working-tree**, so concurrent agents stomp each other. Two concrete failures: **(a)** a broad `git add Docs/web/` (or `git add -A`) in the shared clone sweeps a SIBLING agent's *in-flight* `AUDIT_*`/parity docs into the wrong PR (the RatioRealm incident 2026-07-17: its hub-doc commit swept in equationquest + numbersense AUDIT docs the sibling agents had just Written into the shared tree; caught + fixed via amend + force-push, but only because the agent re-checked its PR file list); **(b)** concurrent `git checkout -b` from two agents in one clone flip the shared HEAD out from under each other. **FIX (belt + suspenders):** give each parallel subagent — AND the orchestrator when it edits rules/docs *mid-flight while subagents run* — its OWN hub worktree off `origin/main` (`git worktree add /tmp/wt-hub-<purpose> origin/main`) for hub-side commits, AND still pathspec-`git add` only its exact `Docs/web/<app>/…` files + its own CLAIMS line (**never** `git add Docs/web/` or `-A` — discipline 6). `.claude/CLAIMS.md` is `merge=union` so per-agent claim lines reconcile; the per-app `Docs/web/<app>/` subtrees are disjoint so per-worktree branches merge clean. This is discipline 6's pathspec rule + `R-SITE-WORKTREE`, extended to the HUB repo under parallel subagents.
   - **⚠ Two MAIN sessions on the SAME lane derive the SAME hardcoded worktree path (`/tmp/wt-<lane>`) → a CREATE/REMOVE collision; add a session-unique suffix, and NEVER `git worktree remove --force` a path you didn't create THIS session even when it's a SPECIFIC (non-glob) path — it may be a live sibling's active worktree (2026-07-18, Logic-hotseat duplicate sessions).** When the founder launches two sessions on the same prompt (e.g. "do the logic clones"), both compute the same lane-named path (`/tmp/wt-logic-hotseat`) — so a resuming session's `git worktree remove /tmp/wt-logic-hotseat --force && git worktree add …` **unregisters + re-checks-out the SIBLING's ACTIVE worktree** (their in-progress edits get re-applied by their still-running process into your fresh checkout — a two-writers-one-tree mess, and the git registration flips to your branch). Fix: **derive the path with a per-session-unique suffix** (`/tmp/wt-<lane>-<pid>` or `-back`/`-front` for a disjoint split) so two sessions never share it; and treat `git worktree remove --force <specific-path>` as destructive-to-others unless YOU created that exact path THIS session (a bare `git worktree prune` is safe; a `--force` remove of a path you didn't just create is not). This is the same-path sibling of the glob-remove hazard below.
   - **⚠ Worktree CLEANUP is as lane-scoped as worktree creation — NEVER `git worktree remove --force` via a broad `wt-*` glob; remove ONLY your own exact paths (2026-07-17).** Sibling sessions use the SAME `/tmp/wt-expand-<app>` / `/tmp/wt-hub-*` naming, so a close-out sweep like `for w in $(git worktree list | grep wt-expand); do git worktree remove "$w" --force; done` deletes a **sibling's active worktree** — and `--force` removes it even with uncommitted changes (their in-flight edits in that dir are lost; the branch + pushed commits survive, unpushed edits do not). Remove your worktrees by their EXACT app path (`git worktree remove /tmp/wt-expand-<your-app> --force`), one per clone you built; a bare `git worktree prune` only clears already-deleted-dir bookkeeping and is safe. Reference: 2026-07-17 Math close-out force-removed a sibling's `wt-expand-weatherforge` via a `wt-expand-*` glob — no committed-work loss (their branch was pushed + PR #888 open at the worktree HEAD SHA), but a real cross-session intrusion. Same lane-discipline as discipline 6 (pathspec, not `-A`): scope every destructive op to your own named targets.

10. **A NON-orchestrator-verifiable JUDGMENT gate makes parallel subagents the WRONG default — prefer main-session serial (founder-directed 2026-07-17).** Some Definition-of-Done gates are **machine-verifiable** (a green `build:play`, `vitest`, `check-site-internal-links`, a schema check) — the orchestrator can re-confirm them from a subagent's output. But the **screenshot-DoD** (`R-WEB-CLONE-SCREENSHOT-DOD`) is a **visual JUDGMENT the orchestrator never re-performs** when a subagent runs it: the subagent reads the pixels, the orchestrator only reads its *written verdict* + the green automated gates. So for **screenshot-DoD-gated work — web-clone builds/spawns AND expansion passes (ADR-048)** — parallel fan-out trades away the ONE gate you most want your own eyes on, and stacks the storm (§ Stagger) + shared-clone contention (disciplines 8–9) on top. **Default to main-session SERIAL** (the agent that will merge reads every dark+light×desktop+mobile capture itself), one clone at a time; use ≤1–2 subagents only when the founder explicitly trades quality-control for throughput on a bounded, templated wave. **The test:** *"can I, the orchestrator, re-verify this DoD gate from the delegate's output alone?"* — YES (build/test/lint/schema) → safe to delegate/parallelize; NO (visual/UX/pixel judgment, subjective quality) → do it in-session-serial (or personally re-render + read the delegate's surfaces before merging). Founder rationale (chose main-session-serial over 2–3 parallel subagents for the Math-cluster expansion, after `numbersense`/`ratiorealm`/`equationquest` shipped correctly via subagents but the storm-risk + un-seen-screenshots concern outweighed the throughput): *"screenshot-DoD is a visual judgment call."* Reserve parallel subagents for read-heavy/mechanical/machine-verifiable work (audits, deep-reads, code-search, doc stubs). See memory [[prefer-serial-for-visual-judgment-gates]].
   - **Concrete rate-limit datapoint reinforcing this (SCIENCE-cluster expansion, 2026-07-18):** launching **3 concurrent** full expansion subagents (each ~450–490K tokens / ~25 min) **rate-limited 2 of the 3** ("Server is temporarily limiting requests · Rate limited") after ~60 tool calls each — only 1 finished; the other two did local work but pushed nothing. Re-run **SERIAL (1 subagent at a time)** shipped 6/6 cleanly with zero rate-limits. So even the ≤1–2-subagent escape hatch is generous for *heavy* expansion agents — treat **1-at-a-time as the default** for expansion/build passes, not 2–3. (Founder's standing warning: *"watch out for rate limits when launching multiple agents."*)
   - **Finishing a serial subagent whose PUSH was declined/interrupted (2026-07-18):** a subagent that completes all build+gates but fails at the push/PR step leaves a **complete, gated local commit** on its worktree branch — do NOT re-run it (wasteful) or restart from scratch. The orchestrator finishes the ship directly: `cd <worktree> && git fetch origin main && git rebase origin/main`, then push+PR+merge. ⚠ **Before the rebase, a `git diff origin/main..HEAD` looks CONTAMINATED with many OTHER clones' files** — that is NOT a pathspec leak; it is the branch being *behind* the sibling merges that landed during the (~25-min) build (the local `origin/main` the worktree branched from is now stale). **The rebase onto FRESH origin/main drops all that noise → the diff becomes correctly scoped to just this clone's files.** Re-run `check-play-css-parse.mjs` + the clone's `vitest` post-rebase (the `play.css` theme-tail union-merge can drop a `}` at the sibling seam), then ship. Reference: weatherforge finished this way after its subagent's push was declined (site PR #888).

### When this rule applies

- Any hub session that MIGHT be running alongside another (assume yes by default — the portfolio routinely has ≥2 concurrent sessions).
- Before any: work-queue/ADR number allocation · portfolio rule sync · R2-mutating op · cross-repo PR merge · sustained per-app or portfolio-wave work.

## Batch portfolio-scale content-distribution pushes — one merge, not one-push-per-app (R-BATCH-DISTRIBUTION-PUSH; 2026-07-10)

**A portfolio-scale content-distribution wave (chapter sync / snapshot+audio regen / any per-app asset
distribution across many apps) MUST NOT push one commit per app to a build-connected `main`.** Batch it:
accumulate the wave on a **`staging/<wave>` branch** and merge to `main` **once** (or squash), turning ~N
pushes into 1. Codified after a 2026-07-10 incident where a V61 content wave pushed **132 one-commit-per-app
syncs** to `spark-anvil-site main` in 24h; because **both** Cloudflare deploy units (core + play) build on
**every** `main` push, that enqueued ~132 × 2 build triggers and saturated the queue to ~100 deep — starving
real `/play` deploys (readquest sat 404 for ~40 min behind the backlog). See
`Docs/AUDIT_CLOUDFLARE_BUILD_QUEUE_SATURATION_2026-07-10.md`.

- **Why it matters:** every push to a build-connected branch triggers a build **per connected unit**. At
  spark-anvil-site there are TWO units, so push count is doubled into the queue, and slow serial core builds
  (~12–20 min) can't drain a one-push-per-app storm.
- **The discipline:** distribution scripts (`sync_content_to_site.sh`, `distribute_*`, `copy_*_to_repos.sh`
  for the SITE repo, wave runners) SHOULD commit per-app locally but **push once at wave end** (staging branch
  → single merge), not push-per-app. For the app-repo direction (140 separate repos) push-per-repo is fine —
  this rule is about **a single build-connected repo** (the site) taking N pushes for N apps.
- **Companion account-level fixes (user-managed, hub can't set):** per-unit build-watch **path filters** (the
  play unit should not build on chapter-only pushes) + **cancel-superseded-builds** — both in the audit doc.

## Combine RELATED, same-lane, non-conflicting work items into ONE PR — don't open N tiny PRs that each trigger their own CI run (R-COMBINE-RELATED-WORK-ONE-PR; 2026-07-21)

**When a single hub agent/session produces MULTIPLE work items that are (a) logically one unit of work, (b) touch the same or non-conflicting files, and (c) would each otherwise open a separate PR, COMBINE them into ONE PR.** Every PR to a build/CI-connected repo (esp. `spark-anvil-site`) triggers at least one CI run (`play-tests`: unit + scope + e2e shard(s) + aggregator = several Ubicloud runners) and one merge cycle — so N tiny related PRs = N× the CI minutes, N× the merge-race/rebase overhead, and N× the runner-pool pressure (the § R-BATCH-DISTRIBUTION-PUSH + `AUDIT_CI_STUCK_RUNS_2026-07-21` pile-up class, at PR granularity). Codified per founder-direct 2026-07-21 (*"codify so that the hub agents combine similar work items into 1 PR"*), after a CI-cost session opened several related PRs (a research doc, its rule fold-in, an audit, two workflow tweaks) that were one logical unit. This is the PR-granularity sibling of R-BATCH-DISTRIBUTION-PUSH (which batches *pushes*); together they cut redundant CI/build triggers at both levels.

**DEFAULT: combine.** A research doc + its rule fold-in; an audit + the fix it documents; several edits to the SAME rule/doc; a cluster of same-surface tweaks discovered together — one PR. Prefer author-all-then-one-PR over drip-feeding a PR per micro-edit.

**EXCEPTIONS — keep SEPARATE (these OVERRIDE the default; combining here is a defect):**
1. **Disjoint per-app `/play` clone builds** stay ONE-PR-PER-APP (`spark-anvil-website.md` § R-PARALLEL-WEB-CLONE-BUILD) — combining couples unrelated deploys + manufactures the merge conflicts that per-app namespacing exists to avoid.
2. **A SHARED-surface edit stays on its OWN PR, separate from per-clone changes** (`spark-anvil-website.md` § R-WEB-CLONE-TEST-PERF + § R-PARALLEL-HUB-AGENTS §5.e) — so a scoped clone PR isn't dragged into the FULL 4-shard sweep, and vice-versa. (Note the interaction: combining several SCOPED clone changes into one PR that ALSO touches a shared file flips the whole PR to a FULL sweep — usually NET-WORSE than the separate scoped runs. Combine within a trigger tier, never across it.)
3. **Cross-repo boundaries** — hub-side docs/rules vs the site/app change are different repos → necessarily different PRs (that is not "N tiny PRs", it's the repo split).
4. **Independent ship cadence / risk** — a risky or experimental change stays isolated for clean rollback; work with different urgency, a different required reviewer, or that unblocks something time-sensitive ships on its own.
5. **A required-check treadmill** — if one item in a combined PR keeps re-conflicting on a hotspot (`play.css`, work-queue, CLAIMS), split THAT item off so it doesn't block the rest (§ R-PARALLEL-HUB-AGENTS §4 escape (a)).

**The test:** *"are these items one logical change, in one repo, touching non-conflicting files, in the same CI-trigger tier (all-scoped OR all-shared, not mixed)?"* — YES → one PR; NO → the exception that applies decides the split. When unsure between one medium PR and three tiny ones, prefer the one PR (fewer CI runs) unless an exception fires. **Cross-refs:** § R-BATCH-DISTRIBUTION-PUSH (push-level sibling) · `spark-anvil-website.md` § R-PARALLEL-WEB-CLONE-BUILD + § R-WEB-CLONE-TEST-PERF + § R-SITE-BUILD-AUTOMATION (CI-cost item) · § R-PARALLEL-HUB-AGENTS §4/§5.e · `Docs/AUDIT_CI_STUCK_RUNS_2026-07-21.md`.

## Back up a long cascade's in-flight gens so a reset never restarts it (R-CASCADE-CHECKPOINT-DURABILITY; 2026-07-11)

**A multi-hour content-regen cascade that commits per-app locally and pushes BATCHED (per R-BATCH-DISTRIBUTION-PUSH) has ALL its unpushed work destroyed the moment the repo is `git reset --hard`'d — and a reset is routinely forced by (a) a `spark-anvil-site` history purge-rewrite (R-SITE-HISTORY-PURGE) or (b) a parallel session's origin advance you must reconcile against.** So "batch the pushes" and "don't lose the work" are in direct tension, and the naive batched cascade loses everything on the next rewrite. Codified after the V61 Wave F closer-narration cascade was reset to zero **three times in one day (2026-07-11)** — ~40 regenerated chapters discarded twice — by the back-to-back V89 media purge + V96 pdf purge + an overlapping parallel V61 session. The two-layer discipline below makes in-flight gens survive any reset with zero re-gen, without violating R-BATCH.

### Layer 1 — reset-proof scratch mirror (outside the git trees)

Mirror the cascade's **small regenerated text artifacts** — chapter snapshot `.md` + sidecar `.beats.json` + `.vtt` — to a scratch dir **outside every git working tree** (e.g. `/Volumes/Data/Backups/<wave>/`), refreshed each monitoring cycle. `git reset --hard` cannot touch it, so after a forced reset you `rsync` the artifacts back = **zero re-gen** (no re-segmentation, no re-TTS). Helper: `scripts/backup_cascade_artifacts.sh` (idempotent rsync; text-only).

- **Do NOT mirror the expensive audio `.m4a`** — it is already durable on R2 (system-of-record, R-R2-SYSTEM-OF-RECORD) and is pruned from `public/` anyway (R-SITE-MEDIA-R2). The `.m4a` is the ONE thing a reset does NOT lose; the cheap text artifacts are what actually vanish.
- **Do NOT mirror the beat `.webp`** — unchanged by an audio-only cascade (beats are reused).

### Layer 2 — clean-gated checkpoint pushes (land progress on origin)

Push held commits to origin **periodically (~every 30 min)**, GATED on a **clean working tree** — skip the push if the tree is mid-app dirty. This bounds loss-on-disruption to one cycle and lands progress on origin, which **survives a subsequent purge** (a history rewrite preserves the current tree). Frequency stays far under the R-BATCH build-queue-storm threshold (~a dozen pushes over a multi-hour run, not per-app).

- **The clean-tree gate is load-bearing:** a monitor that pushes while the cascade is mid-regen (tree dirty) makes `git pull --rebase` abort on the dirty tree → churned/failed pushes (bit 2026-07-11). Test `[ -z "$(git -C <dir> status --porcelain)" ]` before pushing; a dirty tree = skip-this-cycle, never a failure.
- The cascade's own **pass-end `batch_push` remains the PRIMARY push**; checkpoint pushes are the safety net between passes.

### When this applies
- Any multi-hour hub content cascade (chapter snapshot/audio regen, distribution waves) that batches pushes — **especially while `spark-anvil-site` history purges or parallel sessions are active** (the exact conditions that force the resets).
- Pair with: R-BATCH-DISTRIBUTION-PUSH (the batching this protects) · R-SITE-HISTORY-PURGE (the reset trigger) · R-R2-SYSTEM-OF-RECORD (why audio needs no scratch copy) · R-PARALLEL-HUB-AGENTS (claim the lane exclusively so collisions stop) · § "Verify origin state before claiming coverage" (origin ≠ local).

## Architecture Decision Records (MADR convention)

**When a non-trivial decision is made — author a MADR.** Industry-standard Markdown Architecture Decision Record format. Captures the decision + alternatives + consequences in a durable, scannable form so future sessions don't re-litigate.

**When to author**:
- Non-trivial architectural / process / scope choice with alternatives considered
- Reversal of a prior decision (use `supersedes:` front-matter)
- Standing rule that affects future work (e.g., scope-rule changes, methodology rule changes)
- Defer / no-action decision that needs to be documented so future sessions don't re-litigate

**When NOT to author**:
- Tactical fixes (commit messages suffice)
- Routine queue items (those live in the work queue)
- Research artifacts (those go in `RESEARCH_*.md` docs; the MADR may cite the research)

**Format**: `Docs/ADR-NNN_<TITLE_SLUG>.md` with YAML front-matter (`status`, `last-reviewed`, `adr-id`, optional `supersedes` / `superseded-by`). Body sections: `## Status` → `## Context` → `## Decision` → `## Alternatives considered` → `## Consequences` (Positive / Negative / Reversibility) → `## References`.

**Index**: `Docs/ADR_INDEX.md` is the canonical list of ADRs + the template + the relationship between MADR-numbered docs and the historical `DECISION_*.md` docs (ADR-001 through ADR-007 are the 7 pre-MADR `DECISION_*.md` files; ADR-008+ are post-Round-95 MADRs).

**Existing `DECISION_*.md` docs** (7 total, all pre-MADR) are NOT renamed retroactively — they're reclassified as ADR-001 through ADR-007 in the index. Backfilling front-matter on them is a deferred / opportunistic task.

## Cross-Repo Audit Methodology (5 rules — ADR-011)

When authoring or refreshing a cross-repo handoff / portfolio audit, apply these 5 rules jointly. Rules 1-4 codified after Round 96 #487 surfaced that the Round 89 #458 audit mis-classified 3 already-shipped items as OPEN. Rule 5 added Round 112 #532 after Round 109 #529 documented a portfolio-wide finding (R3 "both surfaces" reference impl) in an audit doc that future sessions wouldn't naturally find. See ADR-011.

1. **Pull-first (mandatory)**: `git pull --ff-only` every target repo BEFORE the first grep / read. If `--ff-only` fails on any repo, abort the audit and surface the failure. Stale local clones are the #1 audit failure mode (Round 89 #458 root cause).
2. **Pair-check (1:1 keyword matching)**: for every `HANDOFF_FROM_APP_*.md` (or `HANDOFF_FROM_FORGEKIT_*.md`) found, grep the same repo for sibling responses across BOTH conventions: `HANDOFF_FROM_HUB_*.md` (canonical, 2026-06-11+), `HANDOFF_FROM_SPARK_ANVIL_HUB_*.md` (verbose, 2026-06-11+), AND `HANDOFF_FROM_LABSMITH_*.md` (legacy, pre-2026-06-11). Use keyword-overlap (Jaccard ≥ 0.5), not exact-name match — the `_SHIPPED` suffix convention means filenames diverge. An app-side request is **only** OPEN-NEEDS-HUB-ACTION if it has NO paired sibling response across all three filename forms.
3. **Split-row granularity**: if a request bundles ≥ 2 distinct asks (signaled by "AND" in the filename OR multiple `##` subsections in the body), split the audit row into N sub-rows (e.g., A52a + A52b). Each sub-row independently classified — captures partial-ship state.
4. **Freshness horizon (applies to STATUS + RECOMMENDATION audits)**: every audit doc gets a `freshness-horizon: <N days>` field in YAML front-matter (default 7 days). OPEN rows older than the horizon are auto-flagged "needs-re-verification"; rounds attempting to action them MUST first re-pull + re-pair-check. **Extension (Round 116 #537)**: Rule 4 applies to **recommendation audits** (e.g., ranking audits identifying "next-N pilots" / cluster pilot suggestions / forward-looking candidate lists) NOT just status audits. Before listing items as actionable candidates, the audit MUST verify the recommended artifact doesn't already exist. Empirical evidence: Round 113 found 14 DN orphan PRs already merged (Round 106 #518 audit listed them as remediation candidates); Round 115 found 3 pillar-deepening pilot handoffs already shipped Rounds 82-85 (Round 109 #529 audit listed them as Round 115 candidates). Net session savings: ~8-10h via Rule 4 verify-before-action. **Tooling (Round 128 #553)**: `scripts/recommendation_audit_verify.py` implements the 5-step verify-before-list mechanic at script level. Input: YAML with `audit:` metadata + `candidates:` list (each with `app` + `expected_artifact` + `candidate_class` + `priority`). Output: markdown report classifying ALREADY-SHIPPED / IN-FLIGHT / NEW-CANDIDATE per candidate + actionable-list (NEW-CANDIDATE only) + verify-before-action savings count. Future recommendation audits should pipe candidate lists through this script before listing actionable work.
5. **Audit-to-canonical-propagation**: when an audit yields a portfolio-wide finding — a new pattern, reference implementation, policy clarification, or methodology rule — the round-close MUST include propagation to relevant canonical references BEFORE closing the round. Canonical refs are: (a) `.claude/rules/*.md` (loaded into every CC session), (b) `Docs/TEMPLATE_*.md` (read by implementing sessions), (c) `Docs/DECISION_*.md` / `Docs/ADR-*.md` (canonical policy artifacts). Audit docs alone are insufficient — they decay in visibility, and future sessions read rules/template/decision-doc, not audit-doc history. If portfolio rule sync is needed after a `.claude/rules/*.md` update, run `scripts/copy_rules_to_repos.sh --apply` in the same or immediately following round.

See `Docs/ADR-011_AUDIT_METHODOLOGY_PULL_PAIR_SPLIT.md` for the full rationale + alternatives considered + migration path.

## Verify canonical/consolidated guidance against on-disk GROUND TRUTH — secondary docs drift (R-CANONICAL-DOC-GROUND-TRUTH; 2026-07-15)

**When authoring or consolidating a canonical guidance doc (a playbook / runbook / template / methodology rule) that describes how the repos/code/portfolio are actually structured, VERIFY every load-bearing claim against on-disk reality — read the real repos/files — NOT against the prior secondary docs you're consolidating from. Secondary docs drift from the code, and consolidating from them silently propagates the drift into the new canonical artifact.** Codified per the 2026-07-15 app-spawn-playbook session: `PORTFOLIO_PATTERNS.md` still said *"18-section CLAUDE.md template"* (every real repo is 5-section v2) and *"25 `.claude/rules` files"* (the real set is 29) — both errors were caught ONLY because the playbook was authored by analyzing real repos (fresh AoPS spawns + mature apps via read-only Explore agents), then the drift was reconciled in the same round. Had the playbook been written from PORTFOLIO_PATTERNS alone, it would have shipped the stale numbers as new canon.

- **The discipline:** for any structural claim (file trees, section counts, schema shapes, dependency versions, which files exist), the source of truth is the filesystem/`git` — spot-check it (read the actual file, `grep`, an Explore agent), and where a version is authoritative elsewhere (e.g. ForgeKit → `forgekit/Docs/CHANGELOG.md`), cite that + say "verify current" rather than hardcoding a number that will rot.
- **Reconcile in the same round:** when ground-truth checking surfaces drift in the doc you're consolidating from, FIX the stale source too (don't just avoid copying it) — else the next author re-propagates it. (This session fixed the 3 `PORTFOLIO_PATTERNS` sites.)
- **Companion to** § Cross-Repo Audit Methodology Rule 1 (pull-first) + Rule 5 (audit-to-canonical-propagation) + the freshness/pull-first discipline: those keep *reads* fresh; this keeps *authored canon* true to the code.

### CRITICAL: Pull origin BEFORE freshness queries (in-repo or cross-repo)

When the user asks any "freshness" question — *"any new handoff?"*, *"what changed?"*, *"any open PRs?"*, *"what's the current status?"*, *"what did hub ship?"*, *"cq might have new X"* — the **first tool call must be `git pull --ff-only`** before any filesystem inspection (`ls`, `Grep`, `Read`).

Why: filesystem queries return ground truth for the LOCAL commit, not for origin. Concurrent hub / app sessions ship handoffs + rule updates frequently (5 hub PRs landed during a single CQ session 2026-05-29). Answering a freshness query from stale main produces a wrong answer dressed up as authoritative.

Trigger phrases: "any new", "what's open", "what changed", "what landed", "is there new", "current status", "what's pending", "might have new". When any of these appear with a freshness verb, **pull first, query second**.

If `--ff-only` fails (diverged history), surface that to the user before continuing — don't auto-rebase / auto-merge without acknowledgment.

This is the in-repo analog of the cross-repo Rule 1 in § "Cross-Repo Audit Methodology" + `.claude/rules/portfolio.md` § "Pull Before ANY Cross-Repo Read". Same underlying principle: **never answer a freshness question from stale state**.

Codified 2026-05-29 by CQ session after user-direct: agent answered "any new handoff?" from stale main (2 hub PRs behind) and user had to ask "did you pull origin?" to surface the omission. Lifted to hub canonical R168 #600.

### CRITICAL: Pre-work origin verification (parallel-session collision avoidance)

**Before doing WORK that produces files in an app repo (asset generation, illustration distribution, audio bundles, etc.), `git fetch origin main` for that repo AND verify the target files DON'T already exist on origin.** Codified Round 481 #YYY 2026-06-01 after this session burned ~$5+ on redundant illustration generations where the parallel hub session had already shipped the same content to origin.

**Verification pattern**:
```bash
cd <app-repo>
git fetch origin main 2>&1 | tail -1
# Check if target files already exist on origin
origin_count=$(git ls-tree -r origin/main <target-dir>/ 2>/dev/null | grep -c <pattern>)
[ "$origin_count" -gt 0 ] && echo "SKIP — already on origin" || echo "PROCEED — origin empty"
```

**Why this matters**: when multiple hub sessions race the same trauma-gated cluster work (chapter illustrations / cast portraits / audio dramas), the LATE-arriving session generates redundant assets, hits merge conflicts at PR time, closes its own PR as redundant, AND wastes API spend. The pre-work fetch+check eliminates the race.

**Apply to**:
- Per-app asset generation (`gen_app_illustrations.py --chapters` / `gen_cast_portraits.py` / etc.)
- Per-app handoff doc creation if a sibling hub session may also be authoring
- Bulk distribution scripts (`copy_*_to_repos.sh`) — though these are typically idempotent

**Don't apply to** (these are fine without pre-fetch):
- Hub-local doc edits (rules / ADRs / audits — single session owns these)
- Code changes to hub scripts
- Memory file updates

**Companion to** § "CRITICAL: Pull origin BEFORE freshness queries" above. That rule is about READING origin state; this rule is about WRITING after-verification. Both stem from the same principle: never assume your local view matches origin.

**Implementation hook**: pull-and-check can be added to gen scripts as a default-on flag. Until then, the pattern is human-discipline-driven.

### CRITICAL: Inventory script discipline — use `find`, not multi-glob `ls`

**For per-app filesystem inventory scripts that check file presence across many repos, use `find -maxdepth N -name PATTERN` per check; never use multi-pattern `ls`.** Codified Round 488 2026-06-02 after the `Docs/AUDIT_DOCS_ONLY_APP_RANKING_2026-06-02.md` inventory bug.

**Anti-pattern**:

```bash
cast=$(ls "$d"/Resources/Cast/*.webp "$d"/Resources/Illustrations/cast_*.webp 2>/dev/null | wc -l | tr -d ' ')
```

In zsh under `NO_EXTENDED_GLOB`, when ANY one of multiple glob patterns has no matches, `ls` errors out completely AND outputs nothing → `wc -l = 0`. Result: false-zero report for files that DO exist at the matching path. This bit the R488 docs-only audit by reporting "99/99 missing cast portraits" when the truth was "0/99 missing".

**Canonical**:

```bash
cast=$(find "$d/Resources/Cast" -maxdepth 2 -name "*.webp" 2>/dev/null | wc -l | tr -d ' ')
cast2=$(find "$d/Packages/Libraries/Sources/SharedUI/Resources/Cast" -maxdepth 2 -name "*.webp" 2>/dev/null | wc -l | tr -d ' ')
cast=$((cast + cast2))
```

### CRITICAL: zsh does NOT word-split unquoted `$VAR` — a `for x in $VAR` audit loop silently runs ONCE (R-ZSH-NO-WORD-SPLIT; 2026-07-16)

**The hub shell is zsh, and zsh (unlike bash) does NOT perform word-splitting on an unquoted parameter expansion. So `APPS="a b c"; for x in $APPS; do …; done` runs the loop body EXACTLY ONCE with `$x` = the whole string `"a b c"` — it does not iterate a/b/c.** When that body greps a path built from `$x` (e.g. `grep … "Docs/..._${x}.md"`), the path doesn't exist, grep matches nothing, and the loop **reports a false-empty / false-zero result that looks like a clean audit finding.** This is the sibling gotcha to the `find`-not-`ls` rule above (both are zsh behaviors that silently produce false-empty audit output), and it is load-bearing: in the 2026-07-16 V265 pass-2 audit it made a per-file IA grep report "zero stale-IA tokens" across 18 docs when the grep had actually only run once against a bogus concatenated path — nearly shipping a wrong conclusion until the empty `PlayNarrative` count (known-present) exposed the broken loop.

**The fixes (any one):**
- **A literal word list in the `for`** (most reliable for a fixed set): `for x in a b c; do …; done` — literal words split fine; only *variable* expansions don't.
- **A zsh split flag:** `for x in ${=APPS}; do …; done` (the `${=…}` forces word-splitting), or set an array: `APPS=(a b c); for x in $APPS; do …` (array elements iterate).
- **Pipe + `while read`:** `printf '%s\n' $APPS | while read -r x; do …; done` (with a real newline-delimited source).

**The tell (how to catch a broken loop):** if a `for x in $VAR` loop's per-item output is empty/uniform for items you KNOW are present (a known-present token counts 0, every row identical, or the loop clearly ran once), suspect the no-word-split bug — re-run with a literal list and compare. **Never publish an audit conclusion from a `for x in $VAR` loop without confirming the loop actually iterated** (echo `$x` per pass, or check the item count). Pairs with the spot-check-before-publish companion above and with `R-CANONICAL-DOC-GROUND-TRUTH` (a broken loop is exactly how a "verified against ground truth" claim goes wrong).

### CRITICAL: a wide-char `perl -i` re-encodes the WHOLE file → silent UTF-8 mojibake; use Python / the Edit tool / `perl -CSD` (R-PERL-WIDECHAR-REENCODE; 2026-07-18)

**A `perl -0777 -i -pe` (or any `perl -i`) substitution whose PATTERN or REPLACEMENT contains a codepoint > 255 — an emoji (`\x{1F500}`), a curly quote (`\x{2019}`), an em dash (`\x{2014}`), OR the literal UTF-8 byte of one — silently CORRUPTS every PRE-EXISTING multi-byte UTF-8 sequence ELSEWHERE in the file into double-encoded mojibake.** Mechanism: with no UTF-8 layer, perl reads the file's bytes as Latin-1, but the wide-char literal upgrades the working string to "wide," so on `-i` write perl UTF-8-ENCODES the whole thing — double-encoding the file's untouched `é`/`—`/`·`/`💡` into `Ã©`/`â€"`/`Â·`/`ð Ÿ…`. **The edited region looks correct** (its bytes are freshly written), so the damage is invisible where you're looking; it lands on untouched pre-existing UTF-8. **It passes EVERY automated gate** — build, css-parse, link-check, Vitest — because none inspect rendered glyphs; it surfaces ONLY when you LOOK at the rendered pixels. This is the encoding sibling of § R-ZSH-NO-WORD-SPLIT / § "use `find`, not multi-glob `ls`": a shell/edit idiom that silently produces wrong output dressed up as success.

- **The fix:** for any multi-file edit that inserts non-ASCII OR touches a file that already contains non-ASCII, use **Python** (`io.open(f, encoding="utf-8")` — native UTF-8, explicit `.replace()`) or the **Edit tool** (encoding-safe). If you MUST use perl, `perl -CSD -i` (the `-CSD` flags decode input + encode output as UTF-8, so pre-existing UTF-8 survives and `\x{}` inserts encode correctly). **NEVER a bare wide-char `perl -0777 -i -pe`.**
- **The tell / detection:** after such an edit, grep for the double-encode signatures (`grep -lE $'\xc3\x82\xc2|\xc3\xa2\xc2\x80|\xc3\xb0'` for `Â`/`â€`/`ð`), or `cat -v` a known-good line and confirm the bytes; and **never trust "the insertion looks right"** — the corruption is elsewhere in the file.
- **The load-bearing lesson:** this is why the screenshot-DoD (`R-WEB-CLONE-SCREENSHOT-DOD`) is load-bearing for **text/register correctness**, not just layout — a "renders + zero-console-error" e2e is blind to mojibake; only reading the pixels catches it. Reference: 2026-07-18 VA&D web-clone pass-2 — a `\x{1F500}` Mixed-practice-link insertion corrupted 6 clone landings' pre-existing emoji/dashes/middots; caught by the effectsforge dark screenshot (`â€`/`Â·`/`ð ¡`), fixed by `git checkout origin/main -- <landing>` + re-editing via Python.

### CRITICAL: a Python **implicit string concatenation** in a uniform-edit script silently inserts a stray character — and `esbuild`-parse PASSES it (only the runtime/mount gate catches it) (R-PYTHON-IMPLICIT-STRCONCAT; 2026-07-20)

**In Python, adjacent string literals with NO comma between them CONCATENATE (`("a" "b")` == `"ab"`, NOT a 2-tuple). A uniform-edit script that MEANT to build a tuple/list of insert-lines but omitted the commas gets a single STRING, and then indexing it (`X[0]`/`X[1]`) returns individual CHARACTERS — so a script that appends `X[0]+X[1]` inserts the first two characters of the joined string (e.g. `"\n"`+`"i"` → a stray `i` line) instead of the intended lines.** This is the code-generation sibling of § R-ZSH-NO-WORD-SPLIT / § R-PERL-WIDECHAR-REENCODE / § "use `find`, not multi-glob `ls`": a language idiom that silently produces wrong output dressed up as success.

- **The load-bearing companion — the parse gate is BLIND to it.** The stray insert is *syntactically valid* (a bare `i` on its own line is a legal expression statement; the missing imports leave `renderCameoCard`/`CLONE_CAMEO` as *undefined references*, a RUNTIME `ReferenceError`, not a parse error). So `esbuild.transformSync` / a `check-*-parse` gate PASSES it — "ALL N PARSE OK" is NOT proof the edit is correct. It surfaces only at **runtime**: the island errors (`ReferenceError: i is not defined`) and never mounts. This is exactly why the mount/interaction step of the screenshot-DoD (R-WEB-CLONE-SCREENSHOT-DOD) is a load-bearing gate — it caught a bug that esbuild-parse + mojibake-scan + css-parse all passed. A "parses green" uniform edit is not verified until a surface that USES the inserted code actually runs.
- **The fixes:** (a) build multi-line inserts with an **explicit list + `"".join([...])`** (or a triple-quoted string), NEVER adjacent-literal-in-parens + integer indexing; (b) after a uniform edit, **read back 1–2 edited files and grep for the intended tokens** (`grep -c "import { renderCameoCard"`) + scan for a stray one-char line (`awk '/^[a-z]$/'`) before trusting "N wired"; (c) for any edit whose product is executed, gate on a real **build + mount/run**, not just a syntax parse.
- **Reference:** 2026-07-20 Science DN-S cameo wiring — `IMPORT_ADD=("...cameo';" "\n...castCameos';")` (no comma) → one string; `IMPORT_ADD[0]+IMPORT_ADD[1]` → `"\n"+"i"` → 8 clones got a stray `i` + no cameo imports; esbuild-parse said "ALL 44 PARSE OK"; the screenshot-DoD mount step caught `ReferenceError: i is not defined`; fixed via `"".join(...)`, rebuilt + re-captured clean.

## File Management: MCP vs Filesystem Tools

| Task | Use This | NOT This | Why |
|------|----------|----------|-----|
| Create/edit app target files (.swift) | `XcodeWrite` / `XcodeUpdate` | `Write` / `Edit` | Filesystem writes are invisible to Xcode — files won't be added to the xcodeproj or compiled |
| Create/edit SPM package files (.swift) | `Write` / `Edit` | — | SPM auto-discovers source files from directory structure; no project membership needed. `XcodeWrite`/`XcodeUpdate` also work but aren't required |
| Read source files | `XcodeRead` | `Read` | Uses project navigator paths; consistent with other MCP tools |
| Search code | `XcodeGrep` | `Grep` | Searches within project structure |
| Find files | `XcodeGlob` | `Glob` | Finds files in Xcode project structure |
| List directory | `XcodeLS` | `ls` | Shows Xcode project organization |
| Create directories | `XcodeMakeDir` | `mkdir` | Creates Xcode groups, not just filesystem directories |
| Delete files | `XcodeRM` | `rm` | Removes from Xcode project AND filesystem |
| Move/rename files | `XcodeMV` | `mv` | Updates Xcode project references |

**App target** = files under `[AppName]/` listed in the xcodeproj. **SPM package** = files under `Libraries/` managed by `Package.swift`. Since nearly all source code lives in `Libraries/`, filesystem tools (`Write`/`Edit`) work for most file operations.

**Always use filesystem tools for**: non-source files (`Docs/*.md`, `CLAUDE.md`, `.gitignore`), git operations, shell commands.
